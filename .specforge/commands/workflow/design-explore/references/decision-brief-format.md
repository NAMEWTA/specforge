# 决策简报格式（Decision Brief Format）

> 本参考文档详细说明 design-explore 命令中使用的决策简报格式。

## 何时使用决策简报

在以下场景中使用决策简报格式：
- 审查中发现需要用户决策的问题
- 设计输入信息不足时
- 宪法检查存在冲突需要豁免时
- 评分 < 8 分需要确认改进方案时
- 设计模式选择时

## 格式规范

```
D<N> — <问题标题>
项目/分支: <1 句上下文>
ELI10: <16 岁能懂的白话解释，2-4 句，说明利害>
利害分析：选错的后果（1 句）
推荐：<选项> 因为 <1 句理由>
完整性：A=X/10, B=Y/10（或：注意：选项是类型差异而非覆盖度差异）
优缺点：
A) <选项标签> (recommended)
  ✅ <优点 - 具体可观察，>=40 字符>
  ❌ <缺点 - 诚实，>=40 字符>
B) <选项标签>
  ✅ <优点>
  ❌ <缺点>
总结：<1 句权衡综合>
```

## 字段说明

### D<N> - 决策编号
- 第一次决策使用 D1，后续递增（D2, D3...）
- 这是模型级指令，不是运行时计数器

### ELI10 - 白话解释
- 必须用 16 岁能理解的语言
- 2-4 句话
- 必须说明利害关系（stakes）
- 不要使用技术术语，或首次使用时必须解释

### 完整性评分（Completeness）
- 仅在选项存在覆盖度差异时使用（如：完整实现 vs 快速实现）
- 10/10 = 完整覆盖所有边界情况
- 7/10 = 覆盖主路径
- 3/10 = 快捷方式/捷径

**注意**：如果选项是类型差异而非覆盖度差异（如：选择数据库类型），不要伪造分数，使用：
```
注意：选项是类型差异而非覆盖度差异 — 无完整性评分
```

### 优缺点规则
- 每个选项至少 2 个优点和 1 个缺点（如果是真实选择）
- 每条至少 40 个字符
- 必须具体、可观察，不要泛泛而谈
- 对于单向/破坏性确认，使用：`✅ 没有缺点 — 这是硬性停止选择`

### 推荐标签
- 必须在某个选项后标注 `(recommended)`
- 即使是中立姿态也要标注：`Recommendation: <默认选项> — 这是品味调用，没有强烈偏好`
- AUTO_DECIDE 依赖此标签

## STOP 门禁规则

**每个问题单独调用决策简报，不要批量处理。**

```
**STOP** - 等待用户回应后再继续。不要批量处理此决策。
```

### 何时使用 STOP
- 架构决策（数据模型、系统边界）
- 设计模式选择
- 范围变更
- 宪法豁免
- 任何高风险、不可逆的决策

### 何时不使用 STOP
- 常规编码决策
- 明显的变更
- 用户已明确表达偏好的场景

## 示例

### 示例 1：设计模式选择

```
D1 — 选择设计模式
项目/分支: feature/user-authentication
ELI10: 设计模式决定了审查的严格程度。轻量模式适合小改动，标准模式适合大多数功能，重量模式适合复杂系统。选错会导致要么过度设计要么遗漏关键问题。
利害分析：选错模式会导致设计质量不匹配复杂度，要么浪费时间要么遗漏风险。
推荐：标准模式 因为 它在质量和效率之间取得平衡
完整性：轻量=5/10（覆盖核心但省略边缘）, 标准=8/10（完整审查）, 重量=10/10（全覆盖）
优缺点：
A) 轻量模式
  ✅ 快速完成，适合简单改动，减少不必要的审查开销
  ❌ 可能遗漏边界情况和 DevEx 问题，后续可能需要返工
B) 标准模式 (recommended)
  ✅ 完整三视角审查，覆盖大多数风险，确保设计质量
  ❌ 比轻量模式多花时间，需要更详细的文档
C) 重量模式
  ✅ 最全面，包含性能/安全审查，适合关键系统
  ❌ 耗时长，可能过度设计简单功能，影响交付速度
总结：标准模式适合大多数场景，在质量和效率间取得平衡
```

### 示例 2：设计输入不足

```
D2 — 设计输入信息不足
项目/分支: feature/payment-integration
ELI10: 当前方案缺少关键信息（如支付接口要求、错误处理策略），无法进行完整设计。需要决定是补充信息、基于假设继续、还是返回需求阶段。
利害分析：基于不足信息设计会导致返工或遗漏关键需求，影响支付系统的可靠性。
推荐：补充信息 因为 可以避免后续返工，确保支付系统设计完整
完整性：补充=10/10（完整信息）, 假设=6/10（有猜测成分）, 返回=10/10（但耗时）
优缺点：
A) 补充信息 (recommended)
  ✅ 确保设计基于完整准确的需求，避免后续返工和风险
  ❌ 需要额外时间沟通和确认，可能延迟设计完成
B) 基于假设继续
  ✅ 快速推进，不阻塞进度，可以并行工作
  ❌ 假设可能错误，导致设计返工，支付系统风险高
C) 返回 requirements-clarify
  ✅ 从根本上解决问题，确保需求完整
  ❌ 流程回退，延迟设计完成，影响交付时间
总结：补充信息是风险最低的选择，确保支付系统设计质量
```

## 常见错误

### 错误 1：批量处理多个决策
❌ **错误**：在一个决策简报中列出多个问题
✅ **正确**：每个问题单独使用 D1, D2, D3...

### 错误 2：伪造完整性评分
❌ **错误**：在类型差异的选项上伪造 7/10、8/10
✅ **正确**：使用"注意：选项是类型差异而非覆盖度差异"

### 错误 3：优缺点太短或泛泛
❌ **错误**：`✅ 快速` （太短）
✅ **正确**：`✅ 快速完成，适合简单改动，减少不必要的审查开销` （>=40 字符）

### 错误 4：缺少 ELI10 或利害分析
❌ **错误**：跳过 ELI10 或利害分析行
✅ **正确**：每行都必须存在，这是决策质量的核心

## 自检清单

在发出决策简报前，验证：
- [ ] D<N> 标题存在
- [ ] ELI10 段落存在（包含利害分析行）
- [ ] 推荐行存在且有具体理由
- [ ] 完整性评分（覆盖度）或类型差异说明存在
- [ ] 每个选项有 >=2 个 ✅ 和 >=1 个 ❌，每个 >=40 字符（或硬性停止逃脱）
- [ ] (recommended) 标签在一个选项上
- [ ] 总结行关闭决策
- [ ] 使用工具调用，不是写散文
- [ ] STOP 门禁明确
