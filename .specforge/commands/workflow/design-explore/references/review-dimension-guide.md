# 审查维度评分指南（Review Dimension Guide）

> 本参考文档详细说明 design-explore 命令中三个视角（工程/设计/DevEx）的评分标准和常见问题。

## 评分规则通用标准

### 分数含义

| 分数 | 含义 | 显示规则 |
|------|------|---------|
| 9-10 | 已通过阅读具体代码/文档验证。发现具体问题或优秀实践。 | 正常显示 |
| 7-8 | 高置信度模式匹配。非常可能正确。 | 正常显示 |
| 5-6 | 中等置信度。可能是误报。 | 显示警告："中等置信度，请验证" |
| 3-4 | 低置信度。模式可疑但可能没问题。 | 仅包含在附录中 |
| 1-2 | 推测。 | 仅在严重性为 P0 时报告 |

### 改进路径

当评分 < 8 分时：
1. **明确指出不足**：具体说明哪个维度不够好
2. **提供改进建议**：至少一个可操作的改进方案
3. **使用决策简报格式**：与用户确认改进方向
4. **更新设计**：根据用户反馈调整设计文档

## 工程视角（Engineering Review）

### 维度详解

#### 1. 架构合理性
**检查点**：
- [ ] 分层是否清晰？（表现层/业务层/数据层）
- [ ] 职责是否单一？（每个模块只有一个变化原因）
- [ ] 依赖方向是否正确？（高层不依赖低层实现）
- [ ] 是否有循环依赖？

**常见问题**：
- 业务逻辑泄漏到控制器/路由层
- 工具类变成上帝对象（God Object）
- 循环依赖导致无法独立测试

**评分示例**：
- 9-10 分：已读取具体代码，确认分层清晰，无循环依赖
- 7-8 分：基于设计文档判断，架构模式合理
- 5-6 分：某些模块职责不够清晰，需要验证

#### 2. 数据流设计
**检查点**：
- [ ] 数据从哪里来？（输入源）
- [ ] 数据经过哪些处理？（转换链）
- [ ] 数据到哪里去？（输出目标）
- [ ] 是否有清晰的数据流图？
- [ ] 是否有数据验证点？

**常见问题**：
- 隐式数据流（全局变量、单例）
- 缺少数据验证导致脏数据传播
- 异步流缺少错误处理

#### 3. 边界与接口
**检查点**：
- [ ] 模块边界是否明确？
- [ ] 接口定义是否完整？（请求/响应/错误）
- [ ] 是否有接口契约文档？
- [ ] 版本控制策略是否清晰？

#### 4. 性能考量
**检查点**：
- [ ] 关键路径性能估算？
- [ ] 瓶颈是否识别？
- [ ] 缓存策略是否合理？
- [ ] 是否有 N+1 查询风险？
- [ ] 大数据量场景是否考虑？

**常见问题**：
- 缺少分页导致内存溢出
- 循环内查询数据库（N+1）
- 缺少缓存策略导致重复计算

#### 5. 错误处理
**检查点**：
- [ ] 失败模式是否梳理完整？
- [ ] 降级策略是否明确？
- [ ] 重试策略是否合理？（重试次数、间隔、幂等）
- [ ] 错误分类是否清晰？（业务/系统/校验）

**常见问题**：
- 吞掉异常（catch 但不处理）
- 重试非幂等操作导致数据重复
- 缺少降级策略导致雪崩

#### 6. 复杂度评估（新增）
**检查点**：
- [ ] 圈复杂度是否合理？（建议 < 10）
- [ ] 是否有过度设计？
- [ ] 抽象层级是否合适？
- [ ] 配置复杂度是否可控？

**过度设计信号**：
- 为未来需求预留接口（YAGNI 违反）
- 过多的设计模式堆砌
- 配置项超过 10 个

## 设计视角（Design Review）

### 维度详解

#### 1. 组件设计
**检查点**：
- [ ] 组件/模块职责是否单一？
- [ ] 接口是否简洁？（参数 < 5 个）
- [ ] 组件是否可独立部署/测试？
- [ ] 内聚性是否高？

**常见问题**：
- 组件职责过多（超过 3 个主要职责）
- 接口参数过多（配置对象模式可解决）
- 组件间耦合过紧

#### 2. 交互一致性
**检查点**：
- [ ] API 风格是否一致？（REST/GraphQL/gRPC）
- [ ] 命名规范是否统一？（camelCase/snake_case）
- [ ] 错误码风格是否一致？
- [ ] 日志格式是否统一？

#### 3. 抽象层级
**检查点**：
- [ ] 抽象是否在合适的层级？
- [ ] 是否有抽象泄漏？
- [ ] 高层是否依赖低层实现细节？

**抽象泄漏信号**：
- 业务层知道数据库表结构
- UI 层知道 API 实现细节
- 需要理解底层才能使用高层 API

#### 4. 可测试性
**检查点**：
- [ ] 每个模块是否可独立测试？
- [ ] Mock 成本是否合理？
- [ ] 是否依赖外部状态？
- [ ] 是否有测试桩接口？

**不可测试信号**：
- 依赖全局状态
- 构造函数中创建依赖（而非注入）
- 静态方法调用外部服务

#### 5. 抽象泄漏检测（新增）
**检查点**：
- [ ] 底层实现细节是否泄漏到高层接口？
- [ ] 是否需要理解实现才能使用接口？
- [ ] 接口是否暴露了不必要的数据结构？

**示例**：
- ❌ 泄漏：`processData(dbConnection, sqlQuery)` — 暴露数据库细节
- ✅ 不泄漏：`processData(querySpec)` — 隐藏实现

#### 6. 单一职责检查（新增）
**检查点**：
- [ ] 每个模块是否只有一个变化原因？
- [ ] 修改一个职责是否影响其他职责？
- [ ] 是否可以用一句话清晰描述模块职责？

**测试方法**：
- 如果描述包含"和"（如"处理用户认证和发送邮件"），可能违反 SRP
- 如果需要超过 2 句话描述，可能职责过多

## 开发者体验视角（DevEx Review）

### 维度详解

#### 1. API 易用性
**检查点**：
- [ ] 新人能否在 10 分钟内理解 API 使用方式？
- [ ] 是否有快速开始指南？
- [ ] 是否有示例代码？
- [ ] API 是否自解释？（不需要查文档也能猜对）

**新手测试**：
- 给一个不熟悉项目的人，看他能否在 10 分钟内完成第一个 API 调用
- 如果需要解释超过 3 次，说明 API 不够直观

#### 2. 调试友好度
**检查点**：
- [ ] 出错时能否快速定位问题？
- [ ] 日志是否清晰？（包含上下文、时间戳、请求 ID）
- [ ] 错误信息是否指向具体问题和解决方案？
- [ ] 是否有调试工具或端点？

**好的错误信息示例**：
```
❌ "Internal Server Error"
✅ "支付失败：信用卡过期（卡号尾号 1234）。请更新卡片信息后重试。错误码：PAY-4001"
```

#### 3. 文档完整度
**检查点**：
- [ ] 接口文档是否自解释？
- [ ] 示例代码是否充分？（至少覆盖主路径 + 1 个边界情况）
- [ ] 是否有常见问题解答（FAQ）？
- [ ] 文档是否与代码同步？

#### 4. 新手上手时间估算（新增）
**检查点**：
- [ ] 新开发者需要多长时间才能开始贡献代码？
- [ ] 是否有 onboarding 指南？
- [ ] 本地开发环境搭建是否 < 30 分钟？
- [ ] 是否有代码结构说明？

**评分参考**：
- 9-10 分：< 1 天即可贡献第一个 PR
- 7-8 分：1-3 天
- 5-6 分：1 周
- < 5 分：> 1 周（需要改进）

#### 5. 错误信息清晰度（新增）
**检查点**：
- [ ] 错误信息是否指向具体问题和解决方案？
- [ ] 是否包含错误码（便于搜索）？
- [ ] 是否包含上下文信息（如哪个参数出错）？
- [ ] 是否提供下一步行动建议？

## 开发者 Persona 概念

考虑三种开发者类型：

### 新手（First-time Contributor）
- **特征**：首次接触项目，不熟悉架构和约定
- **需求**：清晰的 onboarding、示例代码、快速反馈
- **设计目标**：10 分钟内完成第一个 API 调用，1 天内贡献第一个 PR

### 常规使用者（Regular User）
- **特征**：熟悉项目，日常开发
- **需求**：一致的 API、清晰的文档、高效的调试工具
- **设计目标**：无需查文档也能猜对 API 用法

### 专家（Deep Contributor）
- **特征**：深度参与，理解内部实现
- **需求**：灵活的扩展点、高级配置、性能优化工具
- **设计目标**：能够优化和扩展系统

**最低要求**：设计必须至少满足常规使用者的需求
**理想目标**：新手也能快速上手

```markdown
### 审查结果汇总

| 视角 | 平均评分 | 关键发现数 | 需用户决策数 |
|------|---------|-----------|-------------|
| 工程 | 7.5/10  | 3         | 2           |
| 设计 | 8.2/10  | 1         | 0           |
| DevEx| 6.8/10  | 4         | 3           |

**HARD-GATE 检查**：
- ✅ 所有视角 >= 6 分
- ⚠️ DevEx 视角 6.8 分（6-7 分范围）→ 记录风险，用户确认继续

**关键风险点**：
1. 错误信息不够清晰（DevEx 2.2）— 建议添加错误码和上下文
2. 新手上手时间预估 > 3 天（DevEx 4）— 建议添加 onboarding 指南
3. 缺少快速开始示例（DevEx 3）— 建议补充示例代码

**用户确认**：[ ] 确认接受风险并继续
```

## 常见审查陷阱

### 陷阱 1：过度关注技术细节，忽略用户体验
**信号**：工程视角 9 分，DevEx 视角 5 分
**解决**：重新平衡，优先考虑开发者体验

### 陷阱 2：评分虚高，没有具体问题支撑
**信号**：所有维度都是 8-9 分，但没有列出具体证据
**解决**：要求每个 >= 8 分的维度必须列出具体证据

### 陷阱 3：发现问题但不提供改进建议
**信号**：评分 < 8 分，但没有说明如何改进
**解决**：必须提供至少一个可操作的改进方案

### 陷阱 4：批量处理多个问题
**信号**：在一个决策中列出 3+ 个问题
**解决**：每个问题单独使用决策简报格式
