# 多视角并行审查（Multi-Perspective Review）

> 每个视角作为独立子代理，并行返回 JSON 结构，再由主代理聚合形成总评。

## 1. 子代理调度协议

### 1.1 派发参数

主代理向每个子代理派发：

- **scope**：被审查的设计文档片段（DESIGN.md 全文或指定段落）
- **viewpoint**：`ceo | engineering | design | devex`
- **context**：相关 PROPOSAL.md / 已存在的宪法 / 项目上下文

### 1.2 返回 JSON 结构

```json
{
  "viewpoint": "ceo",
  "scores": {
    "战略对齐": 8,
    "范围合理性": 7
  },
  "averageScore": 7.5,
  "strengths": ["核心用户场景明确"],
  "gaps": ["未对最坏情况下的回滚做评估"],
  "criticalIssues": [
    {"id": "C-1", "summary": "...", "evidence": "DESIGN.md:120-145"}
  ],
  "recommendation": "REWORK",
  "decisionBriefs": [
    {"id": "D-1", "title": "...", "options": ["A", "B"]}
  ]
}
```

`recommendation` 枚举：`ACCEPT | REWORK | EXPAND_SCOPE | HOLD`

### 1.3 并行规则

- **独立性**：各视角不得互相阅读对方的中间结论
- **并行**：4 个子代理同时派发，最大化覆盖速度
- **超时**：单个子代理 ≤ 5 分钟（壁钟）；超时主代理标记为 INCONCLUSIVE
- **聚合**：主代理在所有子代理返回后构造汇总表，并按 CEO → 工程 → 设计 → DevEx 顺序展示

## 2. 各视角详细问题清单

### 2.1 CEO（Strategic）

| 维度 | 提问清单 |
|------|---------|
| 战略对齐 | 这个方案在解决"用户最痛"还是"团队最爽"？什么场景下用户会主动推荐它？ |
| 范围合理性 | 当前方案是 1x、3x 还是 10x 方案？有没有更小的版本能验证假设？ |
| 价值密度 | 同样投入下，是否存在 ROI 更高的方案？我们有没有偷懒选了"看起来正确"的路径？ |
| 风险/不可逆性 | 哪些决策一旦实施就难以回退（DB schema、API 契约、数据迁移）？ |
| 隐含前提 | 方案默认成立的"假设"是什么？如果它们被推翻，方案是否还成立？ |

**模式选择启发**：
- 团队反复"加功能"信号 → SCOPE REDUCTION
- 用户反馈集中在"还能再做什么" → SELECTIVE EXPANSION
- 竞品已用 10 倍的方式解决 → SCOPE EXPANSION
- 现有方案运行良好但需迭代 → HOLD SCOPE

### 2.2 Engineering

| 维度 | 提问清单 |
|------|---------|
| 架构合理性 | 分层方向是否单向？有无反向依赖？是否引入隐式全局状态？ |
| 数据流设计 | 写路径与读路径是否对齐？事件顺序保证？幂等性？ |
| 边界与接口 | 模块边界是否能用一句话说清？接口是否最小化？ |
| 性能 | 关键路径 P99 估算？热点资源（DB / cache / network）使用模式？ |
| 错误处理 | 失败模式表是否覆盖所有外部依赖？降级策略是否可执行？ |
| 复杂度 | 是否过度设计（YAGNI）？是否欠设计（关键路径无抽象）？ |

### 2.3 Design

| 维度 | 提问清单 |
|------|---------|
| 组件设计 | 每个模块只有一个变化原因吗？接口是否泄漏实现细节？ |
| 交互一致性 | 同类操作的命名、参数顺序、错误形状是否一致？ |
| 抽象层级 | 抽象是否在合适层级？跨层级调用是否有理由？ |
| 可测试性 | 是否能在不启动外部依赖的情况下单元测试？测试需要哪些 fixture？ |
| 抽象泄漏 | 上层是否需要"了解"下层细节才能正确使用？ |
| 单一职责 | 任意组件改动会影响多少个用户故事？ |

### 2.4 DevEx

| 维度 | 提问清单 |
|------|---------|
| API 易用性 | 新人 10 分钟内能否独立调用？默认值是否合理？ |
| 调试友好度 | 出错时报错信息是否指向问题而非现象？日志结构化吗？ |
| 文档完整度 | 接口注释是否说明"为什么"而非"做什么"？示例覆盖最常见 3 个场景？ |
| 上手时间 | 新开发者需要多长时间能贡献第一个 PR？阻塞点在哪？ |
| 错误信息 | 是否给出可执行的修复路径（"运行 X / 检查 Y / 阅读 Z"）？ |

## 3. 聚合规则

主代理在 `Step 2.5 审查结果汇总` 中应用以下规则：

1. **CEO 优先级**：如 CEO `recommendation` 为 `EXPAND_SCOPE` 或 `SCOPE REDUCTION`，必须先与用户对齐范围，**不要**继续展开下游问题，否则下游分析容易作废。
2. **关键发现合并**：来自不同视角但 evidence 重叠的发现合并为一条（避免重复打扰用户）。
3. **决策简报排序**：按"不可逆性 × 影响范围"排序，最不可逆的最先呈现。
4. **WARNING 阈值**：任意视角平均分 6-7 → WARNING；< 6 → HARD-GATE。
5. **共识检测**：3 个及以上视角同时给某维度打低分 → 升级为关键风险，必须 user-confirm。

## 4. 反规避

| 借口 | 现实 |
|------|------|
| "时间紧，跳过 CEO 视角" | CEO 视角恰恰是用最少时间避免大作废的步骤 |
| "团队就一个人，不需要 4 视角" | 同一个人轮换戴 4 顶帽子也比单视角更可靠；切换时间 < 30 分钟 |
| "评分太主观" | 评分是触发"是否需要决策简报"的阈值，不是绝对结论 |
| "并行子代理太复杂" | 即便串行执行 4 视角，也比单视角少漏关键问题 |
