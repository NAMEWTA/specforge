## 1. 文档目标与范围

本文档用于定义 SpecForge 的统一需求基线，确保：

1. 需求完整（功能、非功能、治理、验收）。
2. 结构统一（命令命名、目录模型、阶段模型）。
3. 可落地（可直接指导 CLI 实现与模板生成）。
4. 可追溯（需求—章节—验收映射可回看）。

本文档是后续设计、开发、测试、发布的唯一需求依据。

---

## 2. 总体定位

SpecForge 是一个 AI 原生、规范驱动的开发工作流工具体系：

- CLI 负责生成与管理规范资产。
- 工作流通过文件引用触发（`@.specforge/commands/...`）。
- 命令与技能均使用 Markdown + YAML Front Matter。
- 规则可复用、可演进、可归档。

---

## 3. 核心设计决策（本次优化）

### 3.1 命令命名统一为 `type-name/<type-name>.md`

命令文件名统一采用：

`<type-name>/`（目录） + `<type-name>.md`（目录内主文档）

其中 `type` 表示软件生命周期阶段类型，`name` 表示该类型下的具体命令。

### 3.2 项目级文档根目录统一为 `specforge/`

所有由该体系生成并管理的项目级文档，统一放在项目根下的 `specforge/` 目录内。  
不再使用分散根目录管理。

### 3.3 CLI 实现细化

补充完整 CLI 代码结构、技术栈与版本约束，确保“需求可直接映射实现”。

## 4. 生命周期模型与命令命名规范

## 4.1 生命周期类型（type）

| type | 阶段含义 | 目标 |
|---|---|---|
| `foundation` | 从 0 到 1 的初始化与可行性 | 建立项目基线与边界 |
| `requirements` | 需求澄清与结构化 | 输出可执行需求 |
| `design` | 方案与架构设计 | 输出系统与技术设计 |
| `planning` | 任务拆解与依赖编排 | 输出任务计划与并行策略 |
| `implementation` | 开发实现与代码交付 | 输出代码与阶段总结 |
| `quality` | 测试、审查、修复闭环 | 输出验证报告与修复记录 |
| `release` | 部署、发布、上线与运维移交 | 输出发布/回滚方案 + OPS-RUNBOOK（含部署/监控/告警/依赖/风险） |
| `evolution` | 复盘、沉淀、归档 | 输出经验资产与归档 |

```text
    ├── project.md                      # 项目上下文与概览
    ├── context/                        # 项目知识沉淀（术语/ADR/经验）
    ├── spec/

    ├── changes/
    │   └── <YYYY-MM-DD-ChangeName>/

    │       ├── PROPOSAL.md             # proposal 阶段生成（按需）
    │       ├── Specs/                  # spec 阶段生成（按需，增量规格）
    │       │   └── <capability>.md
    │       ├── DESIGN.md               # design 阶段生成（按需）
    │       └── TASKS.md                # tasks 阶段生成（按需）
    └── archive/
        └── <yyyy-mm>/<YYYY-MM-DD-ChangeName>/
```

## 5.2 目录职责约束

- `.specforge/`：框架能力“系统态”（命令、技能、模板）。
- `specforge/`：项目持久化资产，包含项目上下文（`project.md`）、知识沉淀（`context/`）、规范、变更与归档，更新不覆盖。
- `specforge/spec/`：规范“当前态”，每个规范为一个 `YYYY-MM-DD-SpecName.md` 单文件。
- `specforge/changes/`：执行“进行态”，每个 change 为 `<YYYY-MM-DD-ChangeName>/` 并包含 `.specforge.json` 状态文件。
- `specforge/archive/`：历史“沉淀态”。

三者共同构成可追踪、可回滚、可审计的文档资产链。

补充边界说明：

- `.specforge/templates`：全局模板库（多命令共享、版本升级可统一维护）。
- `.specforge/commands/<type-name>/templates`：命令私有模板（仅该命令使用）。
- `specforge/context`：项目知识沉淀，不随单次 change 清空，更新不覆盖。
- `specforge/project.md`：项目级上下文与概览，更新不覆盖。

## 6. 统一元数据规范

所有命令/技能/步骤文件必须包含：

```yaml
---
id: <unique-id>
name: <display-name>
type: <workflow-command|domain-rule|...>
description: <short-description>
---
```

可选字段：

- `phases`
- `triggers.keywords`
- `triggers.files`
- `priority` (`high|medium|low`)
- `tech_stack`

## 6.1 `specforge/spec` 单文件规范（参考 brainstorming 结构）

`specforge/spec` 不再采用多层目录；每个规范文档均为单个文件，命名：

`YYYY-MM-DD-SpecName.md`

示例：

- `2026-05-04-NotificationCenter.md`
- `2026-05-12-BillingSettlement.md`

### 单文件 YAML 元数据（必须）

```yaml
---
spec_id: "2026-05-04-notification-center"
title: "Notification Center"
date: "2026-05-04"
status: "draft" # draft | executing | completed
owner: "team-or-user"
source_command: ".specforge/commands/requirements-product-requirements/requirements-product-requirements.md"
---
```

其中 `status` 取值含义：

- `draft`：草稿状态（需求澄清/方案比较阶段）
- `executing`：执行状态（已进入任务实现）
- `completed`：完成状态（通过验收并归档）

### 单文件正文建议结构（参考 brainstorming）

1. **Context Scan**：当前项目背景、相关文件、既有约束
2. **Clarifying Questions**：澄清问题与答案记录（单问题逐步收敛）
3. **Approach Options (2-3)**：方案对比、trade-off、推荐方案
4. **Approved Design**：用户确认后的设计结论（架构/组件/数据流/错误处理/测试）
5. **Execution Scope**：本次实现范围、非目标、风险边界
6. **Spec Self-Review**：占位符检查、一致性检查、范围检查、歧义检查
7. **User Review Gate**：用户评审结论与修改记录
8. **Plan Handoff**：交给实现计划的输入摘要

## 6.2 `specforge/changes` 的 `.specforge.json` 元数据规范

每个 change 目录必须包含 `.specforge.json`，用于记录状态与工作副本定位信息。

```json
{
  "changeId": "2026-05-04-NotificationCenter",
  "status": "draft",
  "phase": "proposal",
  "worktree": {
    "branch": "feature/notification-center",
    "rootPath": "/abs/path/to/worktree",
    "changePath": "specforge/changes/2026-05-04-NotificationCenter"
  },
  "artifacts": {
    "proposal": "PROPOSAL.md",
    "specsDir": "Specs/",
    "design": "DESIGN.md",
    "tasks": "TASKS.md"
  },
  "timestamps": {
    "createdAt": "2026-05-04T11:00:00+08:00",
    "updatedAt": "2026-05-04T11:20:00+08:00"
  }
}
```

`status` 取值：

- `draft`
- `executing`
- `completed`

`phase` 取值：

- `proposal`
- `spec`
- `design`
- `tasks`

## 6.3 `specforge/changes` 产物按需生成规则（OpenSpec 对齐）

`specforge/changes/<YYYY-MM-DD-ChangeName>/` 的文档不是一次性全量生成，而是由命令阶段按需生成：

1. **proposal 阶段**：生成 `PROPOSAL.md`  
   - 作用：锁定意图、范围、方法（Why / What Changes / Capabilities / Impact）
2. **spec 阶段**：生成 `Specs/`  
   - 作用：增量规格，使用 `ADDED/MODIFIED/REMOVED` 表达行为变更  
   - 场景格式：`WHEN/THEN`
3. **design 阶段**：生成 `DESIGN.md`  
   - 作用：技术设计，包含接口定义、数据结构、涉及文件/目录、关键决策与权衡
4. **tasks 阶段**：生成 `TASKS.md`  
   - 作用：粗粒度任务清单与执行顺序

状态联动建议：

- `proposal/spec/design` 阶段默认 `status = draft`
- 进入 `tasks` 并启动实现后切换为 `status = executing`
- 验收通过后切换为 `status = completed`

## 7. 功能需求（FR）

### FR-001 初始化

- CLI 必须支持 `specforge init`，生成第 5 章定义的双根目录结构（`.specforge/` + `specforge/`）。
- 初始化后命令结构必须符合 `type-name/<type-name>.md`（每命令一个目录，根下仅一个主 md）。
- 初始化后 `.specforge/skills` 必须支持“skill 包目录模式”（`<skill-name>/SKILL.md`）并兼容单文件模式。
- 初始化后 `specforge/spec/` 必须采用单文件规范模式（`YYYY-MM-DD-SpecName.md`）。
- 初始化后 `specforge/changes/` 必须采用 `<YYYY-MM-DD-ChangeName>/` 命名并包含 `.specforge.json`。

### FR-002 命令执行

- 必须支持 `@.specforge/commands/<type-name>/<type-name>.md` 触发阶段执行。
- 每个命令执行必须遵循四段协议：准备上下文→执行步骤→生成制品→完成衔接。

### FR-003 技能注入

- 按 `phases`、`triggers` 自动注入技能。
- 冲突按 `priority` 处理。

### FR-004 扩展能力

- `specforge add-skill <name>`：创建技能。
- `specforge add-command --type <type> --name <name>`：创建 `<command-name>/` 命令目录与同名主 md，并按需生成 `scripts/references/templates` 子目录。

### FR-005 资产管理

- `specforge list`：列出 `.specforge/commands` 与 `.specforge/skills`。
- `specforge update`：可更新 `.specforge/` 能力资产，但必须保留 `specforge/` 下所有用户资产（`project.md`、`context/`、`spec/`、`changes/`、`archive/`）。
- `specforge/spec` 下规范文件状态必须由 YAML `status` 字段管理（`draft | executing | completed`）。
- `specforge/changes` 下 change 状态与 worktree 定位必须由 `.specforge.json` 管理。
- `specforge/changes` 下工件必须按阶段按需生成：`PROPOSAL.md` → `Specs/` → `DESIGN.md` → `TASKS.md`。

### FR-006 治理护栏

- 必须支持老项目护栏（B1-B6）与高风险中断确认机制。

## 8. 非功能需求（NFR）

1. **可移植性**：支持 macOS/Linux/Windows。
2. **离线可用性**：核心命令不依赖网络。
3. **轻量性**：初始化与列表操作应快速完成。
4. **上下文可控**：渐进式披露，避免一次性注入过载。
5. **可维护性**：版本演进兼容用户已有资产。
6. **可审计性**：需求、设计、实现、测试、复盘链路可追踪。

## 9. 治理护栏（GR）

- **B1 入场扫描**：技术栈/抽象盘点。
- **B2 架构对齐**：明确复用与新引入边界。
- **B3 边界约束**：read/write 范围声明与越界阻断。
- **B4 破坏性门槛**：公共接口/大删改先引用图再确认。
- **B5 复用优先**：先检索再新增。
- **B6 视觉语汇对齐**：前端场景强制视觉一致性检查。

## 12. 验收标准（更新后）

- [ ] 命令目录与主文件命名全部符合 `type-name/<type-name>.md`。
- [ ] `.specforge/` 仅承载框架能力资产（commands/skills/templates）。
- [ ] `.specforge/skills` 支持 `<skill-name>/SKILL.md` 目录模式，并兼容 `<skill-name>.md`。
- [ ] 项目持久化资产仅落在 `specforge/` 根下（`project.md`、`context`、`spec`、`changes`、`archive`）。
- [ ] `specforge/spec` 不含子目录，规范文档均为 `YYYY-MM-DD-SpecName.md` 单文件。
- [ ] 每个规范单文件都包含 YAML `status`，且状态值仅为 `draft|executing|completed`。
- [ ] `specforge/changes` 下目录命名均为 `<YYYY-MM-DD-ChangeName>/`。
- [ ] 每个 change 目录都包含 `.specforge.json`，并含 `status/phase/worktree` 定位字段。
- [ ] change 工件按命令阶段逐步生成（非一次性全量预生成）。
- [ ] `Specs/` 使用 ADDED/MODIFIED/REMOVED，场景遵循 WHEN/THEN。
- [ ] `specforge init` 生成 `specforge/spec`、`specforge/changes`、`specforge/archive`。
- [ ] `add-command` 能生成 `<command-name>/` 目录及同名主 md（可选子目录按需创建）。
- [ ] CLI 技术结构与版本清单可直接指导实现与发布。
- [ ] 旧项目迁移时可平滑从旧目录结构映射到新结构。

## 13. 结论

本次优化后，需求文档已明确三项关键标准：

1. **命令目录标准化**：统一 `type-name/<type-name>.md`，每命令目录可按需挂载脚本与参考资源。
