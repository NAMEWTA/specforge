# 长跑健康守则（Phase 4 配方）

> 在 Phase 4 加载本文件。本文件给出"如何让 AI 长时间健康地跑完一件事"的具体配方——心跳、检查点、防漂移、反重复、预算守门、上下文重置。

## 1. 为什么要专门写"健康守则"

短任务（5-15 分钟）能靠模型本身的注意力撑下来。长任务（半小时以上、跨多个工具调用、上下文逼近窗口上限）会出现一类典型病：

| 病 | 症状 | 根因 |
|----|------|------|
| **漂移** | 改着改着开始动 Scope 之外的文件 | 没有显式自检机制 |
| **死循环** | 同一报错一改再改 | 没有"停下汇报差异"约束 |
| **假完成** | 宣布完成实际跑都没跑 | 验收依赖模型自觉 |
| **预算耗尽** | 突然 token 用尽中断 | 没有预算守门 |
| **上下文被噪音淹没** | 后半段只看到 stdout 末尾 | 没有 context reset 触发条件 |

健康守则是为了把这五种病在 goal 里就堵住。

## 2. 五条核心守则（按顺序注入）

### 2.1 Heartbeat（心跳）

**配方**：

```
每完成一项 Done when 子项后，输出一条心跳：
  - 当前在哪一项（Done when 第 N 条）
  - 已完成什么（修改文件 list / 新增测试名）
  - 剩余预算估算（已用 token / 总预算）
  - 是否仍在 Scope 内
```

**注入位置**：goal 末尾"健康守则"段。

**反模式**：

- ❌ "做完所有事再汇报"——长跑一旦失联，回滚成本巨大。
- ❌ 心跳里只说"在做 X"，没数字、没文件——本质是无信息回报。

### 2.2 Checkpoint（检查点）

**配方**：

```
每完成 1-2 项 Done when 子项立即 commit。
commit message 用 conventional commits：feat / fix / refactor / test / docs / chore。
单次 commit 不超过 200 行 diff、不超过 5 个文件——超过即拆分。
禁止"全做完再 squash"——这会让回滚边界消失。
```

**注入位置**：Constraints 末尾 + 健康守则段。

**特殊场景**：

- 场景 D（代码考古）/ E（UI audit）/ F（守门员）：不动代码，但要求每写完一份 docs/*.md 就 commit。
- 场景 C（批量任务）：天然就是"一件事一个 commit"，加强写在 Done when 第 1 条。

### 2.3 Anti-drift（防漂移自检）

**配方**：

```
每次心跳前先自答（回答"否"则停下）：
  - 本次将要修改的文件是否仍在 Scope 内？
  - 是否触碰了 Constraints 中的 MUST NOT modify 清单？
  - 实现路径是否仍然符合 Objective 描述的 after 状态？
任何一个"否"都立即停下汇报，等用户决定是扩 Scope 还是放弃此路径。
```

**注入位置**：健康守则段。

**反模式**：

- ❌ "我觉得改一下这个相邻文件会更优雅"——这就是漂移的开始。Scope 扩张必须用户授权。
- ❌ 心跳只说"还在 Scope 内"，不复述 Scope 文本——形式主义。

### 2.4 Anti-loop（P9 反重复）

**配方（直接照抄到 goal）**：

```
同一根因（同一报错栈 / 同一测试失败 / 同一文件解析错误）连续失败 ≥ 2 次时，必须：
  1. 立即停止重试。
  2. 书面写出已尝试的方案 list（每条 1 行，含失败原因）。
  3. 书面写出"下一条想试的方案"与"已试方案的具体差异"（差异不能仅是参数微调）。
  4. 等用户指令后再继续。
违反此条等同触发 SpecForge 宪法 P9 的 E010_repeatedFailurePattern。
```

**注入位置**：Stop if 段（不放健康守则段——P9 是停止条件而非自检）。

**反模式**：

- ❌ "我换个写法再试一次"（差异仅是格式微调）——这是同根因。
- ❌ "这次一定行"——意图陈述不算差异。差异必须是"涉及不同模块 / 不同抽象层 / 不同根因假设"。

### 2.5 Token budget guard（预算守门）

**配方**：

```
剩余预算 < 20% 时主动汇报：
  - 已完成的 Done when 子项 list
  - 未完成的子项 list
  - 推荐的拆分点（"建议将剩余项拆为新 goal，新 Scope = ..."）
  - 当前修改是否处于"可 commit"状态（如否，先回到上一个 commit 状态）
不要硬撑到 0，硬撑会留下未完成的半成品 commit，回滚成本极高。
```

**注入位置**：健康守则段。**与 Heartbeat 联动**：每次心跳都顺便报一次预算。

## 3. 三条增强守则（按需注入）

### 3.1 Context reset trigger（上下文重置触发）

**何时加**：goal 预算 ≥ 100K，或场景为 B（功能 + spec，文件多）。

**配方**：

```
出现以下任一情况，先 commit 当前进度 + 写 PROGRESS.md，然后建议用户 /compact 或重启：
  - 上下文使用率 > 85%
  - 最近 3 次心跳都在同一项 Done when（卡死信号）
  - 已读文件累计 > 50 个（噪音过多信号）
PROGRESS.md 至少包含：当前在第几项、已完成 list、未完成 list、下一步计划。
```

### 3.2 Tool-output framing（工具输出框架）

**何时加**：goal 涉及大量 stdout 输出（Swift xcodebuild / Rust cargo test / 大型 monorepo build）。

**配方**：

```
工具输出超过 2000 行时，禁止只看末尾。必须：
  1. 用 grep 抓取关键摘要（如 "test result:" / "BUILD SUCCEEDED" / "FAILED"）。
  2. 心跳中复述抓到的摘要（不是抓"看起来通过了"）。
  3. 如关键字未抓到任何匹配，视为工具异常，停下汇报。
```

### 3.3 Heartbeat to file（心跳落盘）

**何时加**：goal 持续时间预计 ≥ 30 分钟，且工具不保留对话历史（如 sub-agent / 一次性脚本）。

**配方**：

```
每次心跳同时追加到 PROGRESS.md（一个 markdown bullet 一行），便于失联后人工接力。
PROGRESS.md 不进 commit（gitignore 或放 /tmp/），避免污染主分支。
```

## 4. 工具差异（与 references/tool-adapters.md 联动）

| 工具 | 心跳形式 | Checkpoint 频率 | Token guard 是否生效 |
|------|---------|---------------|---------------------|
| Codex CLI 0.128+ | 直接 print，被 `continuation.md` 审计循环消费 | 每 1-2 子项 | 是（has token budget native） |
| Claude Code | "请明确回报"形式的指令 | 每 1-2 子项 | 部分（依赖 Sonnet/Opus 自报） |
| Cursor | 写到 chat 即可 | 每 1 子项（Cursor 上下文紧） | 否（人工目测） |
| Kiro Autopilot | 直接 print | 每 1-2 子项 | 是（context awareness 自动 compact） |
| 通用 LLM | 每轮回复尾部固定段落 | 每轮 | 否 |

## 5. 健康守则块的最小可用形态

如果 goal 整体很短（如重构单文件），不必把上面 5 条都塞进去。**最小可用形态**：

```
【健康守则】
- 每完成一项 Done when 子项后：心跳（当前项 / 已改文件 / 剩余预算估算）+ commit。
- 同根因连续失败 ≥ 2 次：停下，书面汇报与已试方案的差异，等指令（P9）。
- 剩余预算 < 20%：停下汇报已完成 / 未完成 + 推荐拆分点。
```

三条覆盖了最常见的三种病。其他守则按场景加。

## 6. 反模式

| 反模式 | 修法 |
|-------|------|
| "保持注意力" / "不要分心" | 改成可机械检测的指令（"每 1-2 项 commit") |
| 把所有守则压成一句话 | 长跑工具会忽略——分点列出 |
| "如果觉得不对就停" | 改成"如果 X 出现就停"（X 可机械检测） |
| 心跳要求"汇报状态"但没说汇报什么 | 显式列出 4 字段（项 / 文件 / 预算 / Scope） |
| 让模型自己估算何时该 reset 上下文 | 给阈值（85% / 3 次 / 50 文件） |
