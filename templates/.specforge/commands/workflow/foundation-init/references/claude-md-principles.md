# CLAUDE.md — 4 条 Karpathy 编码原则

> 本参考文档供 `foundation-init` Step 4 加载，用于在项目根目录新建 / 补齐 `CLAUDE.md`。
>
> 原则来源：Andrej Karpathy 关于 LLM 编码陷阱的观察。模型会代你做错误假设、不假思索执行，不管理自身困惑、不寻求澄清、不呈现矛盾、不展示权衡，喜欢堆抽象、爱触碰理解不足的代码。这 4 条原则是给 Claude / Cursor / Codex / Kiro 等 AI 代理的"硬护栏"。

## 1. 原则总览

| 原则 | 解决的问题 |
|------|-----------|
| **编码前思考** | 错误假设、隐藏困惑、缺少权衡 |
| **简洁优先** | 过度复杂、臃肿抽象、为不会发生的场景做防御 |
| **精准修改** | 无关编辑、触碰不该碰的代码、顺手"改进" |
| **目标驱动执行** | 通过测试优先、可验证的成功标准、独立循环执行 |

## 2. 原则 1：编码前思考（Think Before Coding）

> 不要假设。不要隐藏困惑。呈现权衡。

LLM 经常默默选择一种解释然后执行。这条原则强制明确推理：

- **明确说明假设** — 不确定就问，不要猜
- **呈现多种解释** — 存在歧义时，列出来让用户选，不要默默选一种
- **适时提出异议** — 如果存在更简单方法，说出来，不要顺着用户错的方向走
- **困惑时停下来** — 指出不清楚的地方并要求澄清

**❌ 反例**

> 用户："给登录加个限流"
> AI：（默默假设是 IP 维度限流、默认 60 次/分钟、用 Redis）→ 直接写代码

**✅ 正例**

> 用户："给登录加个限流"
> AI："限流维度是按 IP 还是按用户名？阈值预期是多少？是否已有 Redis 可用？这三个会改变实现方式。"

**What this prevents**：避免 AI 用错误假设构建 1000 行最终被推倒重写的方案。

## 3. 原则 2：简洁优先（Concise First）

> 用最少的代码解决问题。不要过度推测。

对抗过度工程的倾向：

- 不添加要求之外的功能
- 不为一次性代码创建抽象
- 不添加未要求的"灵活性"或"可配置性"
- 不为不可能发生的场景做错误处理
- 如果 200 行代码可以写成 50 行，重写它

**检验标准**：资深工程师会觉得这过于复杂吗？如果是，简化。

**❌ 反例**

> 任务："写一个把 JSON 字符串解析成对象的工具函数"
> AI：写了一个支持 5 种字符编码、3 种容错策略、自定义 reviver 的 Generic Parser，外加 Factory 模式和 Builder 模式

**✅ 正例**

> ```ts
> export function parseJson<T = unknown>(input: string): T {
>   return JSON.parse(input) as T;
> }
> ```
> 如果以后真的需要容错或自定义 reviver，再加。

**What this prevents**：避免堆砌抽象、过早泛化、把 100 行任务实现成 1000 行架构。

## 4. 原则 3：精准修改（Surgical Edits）

> 只碰必须碰的。只清理自己造成的混乱。

编辑现有代码时：

- 不"改进"相邻的代码、注释或格式
- 不重构没坏的东西
- 匹配现有风格，即使你更倾向于不同的写法
- 如果注意到无关的死代码，**提一下**——不要删除它

当你的改动产生孤儿代码时：

- 删除因你的改动而变得无用的导入 / 变量 / 函数
- 不删除预先存在的死代码，除非被要求

**检验标准**：每一行修改都应该能直接追溯到用户的请求。

**❌ 反例**

> 任务："修复 `formatDate` 函数的时区 bug"
> AI：顺手把同文件其他函数的 `var` 改成 `const`、把 4 空格缩进改成 2 空格、删了一段看起来没用的注释

**✅ 正例**

> AI：只改 `formatDate` 函数的时区处理逻辑。在 PR 描述中说："注意到同文件另有 3 处 `var`，但与本次任务无关，未触动；如需统一可单独开 PR。"

**What this prevents**：避免 PR 范围爆炸、merge 冲突激增、code review 失焦。

## 5. 原则 4：目标驱动执行（Goal-Driven Execution）

> 定义成功标准。循环验证直到达成。

将指令式任务转化为可验证的目标：

| ❌ 不要这样写 | ✅ 转化为 |
|--------------|---------|
| "添加验证" | "为无效输入编写测试，然后让它们通过" |
| "修复 bug" | "编写重现 bug 的测试，然后让它通过" |
| "重构 X" | "确保重构前后测试都能通过" |

对于多步骤任务，先说明一个简短的计划：

```
1. <步骤> → 验证：<可观测的检查>
2. <步骤> → 验证：<可观测的检查>
3. <步骤> → 验证：<可观测的检查>
```

强成功标准让 LLM 能够独立循环执行；弱标准（"让它工作"）需要不断澄清。

**❌ 反例**

> 任务："修一下登录有时会失败的问题"
> AI：改了 5 处看起来相关的代码，跑了一下能登录就汇报"已修复"

**✅ 正例**

> AI：
> 1. 写一个能稳定复现失败的测试用例（验证：测试失败）
> 2. 定位根因（验证：在代码中找到具体行）
> 3. 修复（验证：第 1 步的测试转绿，且回归全套测试通过）
> 4. 报告每一步的可观测证据。

**What this prevents**：避免"看起来好了"的伪修复、避免"跑一下能动就当完成"。

## 通用编码原则（基于 Karpathy 观察）

### 1. 编码前思考
- 不要假设；不确定就问，不要猜
- 呈现多种解释，让用户选，不要默默选一种
- 困惑时停下来澄清，不要硬撞
- 如果存在更简单方法，提出异议

❌ 默默选一种解释执行
✅ 列出歧义点，问清楚再写

### 2. 简洁优先
- 用最少代码解决问题
- 不添加要求之外的功能 / 灵活性 / 配置性
- 不为不会发生的场景做错误处理
- 200 行能写成 50 行就重写

❌ 一次性脚本上 Factory + Builder
✅ 5 行函数就 5 行函数

### 3. 精准修改
- 只碰必须碰的
- 不"改进"相邻代码、注释、格式
- 匹配现有风格，即使你更倾向于不同写法
- 注意到无关死代码：**提一下**，不删除
- 删除你自己改动产生的孤儿代码

❌ 顺手把同文件其他函数也"现代化"了
✅ 只改本次任务相关的行

### 4. 目标驱动执行
- 把模糊指令转化为可验证的成功标准
- "修 bug" → "写一个能复现 bug 的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"
- 多步骤任务先列计划，每步配可观测的验证

❌ 改完跑一下"能动"就当完成
✅ 写测试 → 跑测试 → 看到具体证据 → 报告

## 触发条件

每次开始一个编码任务前，AI 必须默念这 4 条原则。如果一项任务无法用这 4 条衡量，先回到 `AGENTS.md` § 5 找答案，仍不清楚就问用户。
```
