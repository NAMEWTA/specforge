# Planning Patterns（计划模式库）

> 本文件汇集了来自 superpowers-zh、spec-kit、claude-task-master 等项目的最佳计划模式，供 planning-breakdown 命令在执行任务拆解时参考。

---

## 模式 1: 文件结构先行

**来源**: superpowers-zh writing-plans  
**适用场景**: 任何需要拆任务的设计阶段

### 核心理念

在拆解具体任务之前，先锁定"哪些文件需要动"，再确定"每个文件做什么"。这是避免任务模糊、确保可执行性的关键步骤。

### 执行步骤

1. 列出所有需要创建/修改/删除的文件
2. 为每个文件标注：操作类型、精确路径、职责描述、测试文件
3. 检查设计原则：单一职责、一起变更的文件放在一起、小而专注

### 文件清单模板

对每个涉及的文件，明确以下信息：

```
- 操作：创建 / 修改（第 N-M 行） / 删除
- 路径：exact/path/to/file.ts
- 职责：一句话描述
- 测试：tests/exact/path/to/file.test.ts
```

### 实际示例

```
- 操作：创建
- 路径：src/services/user-service.ts
- 职责：处理用户相关的业务逻辑（创建、查询、更新）
- 测试：tests/unit/services/user-service.test.ts

- 操作：修改（第 12-45 行）
- 路径：src/models/user.ts
- 职责：扩展 User 接口，添加 emailVerified 字段
- 测试：tests/unit/models/user.test.ts

- 操作：创建
- 路径：src/errors/user-errors.ts
- 职责：定义用户相关的自定义错误类型
- 测试：tests/unit/errors/user-errors.test.ts
```

### 设计原则检查清单

- [ ] **单一职责**：每个文件只有一个明确的职责
- [ ] **共变更原则**：一起变更的文件应该放在一起（按职责拆分，而非按技术层级拆分）
- [ ] **小而专注**：优先选择小而专注的文件，而非承担过多功能的大文件
- [ ] **无重复**：没有两个文件做同一件事
- [ ] **遵循现有模式**：在现有代码库中，遵循已有模式（不要单方面重构）

### 何时允许大文件

- 现有代码库已经使用大文件模式
- 修改的文件已经变得难以管理，需要在计划中包含拆分任务
- 明确的业务理由（如性能优化、减少导入开销）

## 模式 2: 小步骤任务拆解

**来源**: superpowers-zh writing-plans  
**适用场景**: 所有实现类任务

### 核心理念

每个步骤应该是一个 2-5 分钟可完成的操作。这个粒度是经过验证的最优区间——既能保证任务明确可执行，又不会过度琐碎。

### 粒度对照表

| 粒度 | 时长 | 示例 | 评价 |
|------|------|------|------|
| 过粗 | > 15 分钟 | "实现用户模块"（含 8 个文件、15 个函数） | ❌ 无法精确执行 |
| 合适 | 2-5 分钟 | "编写 UserService.createUser 的失败测试" | ✅ 明确可执行 |
| 过细 | < 1 分钟 | "创建 user.ts 文件"（没有实际逻辑） | ❌ 过度琐碎 |

### 标准步骤类型

每个实现任务通常包含以下步骤：

1. **编写失败的测试**（1-2 分钟）
2. **运行测试验证失败**（30 秒）
3. **编写最少实现代码**（2-3 分钟）
4. **运行测试验证通过**（30 秒）
5. **Commit**（30 秒）

### 任务描述要求

- ✅ **正确**：包含精确文件路径、具体操作、预期结果
- ❌ **错误**：模糊描述、缺少文件路径、只说做什么不说怎么做

### 示例对比

```
❌ 错误示例：
- [ ] T005 实现用户创建功能

✅ 正确示例：
- [ ] T005 编写 UserService.createUser 失败测试
  - 文件：tests/unit/services/user-service.test.ts
  - 断言：expect(user.email).toBe('test@example.com')
  - 运行：pnpm test tests/unit/services/user-service.test.ts
  - 预期：FAIL - "createUser is not defined"
```

## 模式 3: TDD 三步节奏

**来源**: superpowers-zh writing-plans  
**适用场景**: 需要高质量代码的实现任务

### 标准 TDD 节奏

对每个实现任务，标注完整的 TDD 循环：

```
- [ ] 步骤 1：编写失败的测试
- [ ] 步骤 2：运行测试验证失败
- [ ] 步骤 3：编写最少实现代码
- [ ] 步骤 4：运行测试验证通过
- [ ] 步骤 5：Commit
```

### TDD 节奏标注模板

在 TASKS.md 中，对每个实现任务使用以下格式：

```
- [ ] T00X [P] [US1] 编写 <场景> 的失败测试 → 运行确认失败 → 实现最少代码 → 运行确认通过 → Commit
  - 测试文件：tests/unit/xxx.test.ts
  - 实现文件：src/xxx.ts
  - 测试命令：pnpm test tests/unit/xxx.test.ts
```

### TDD 优势

1. **防止过度设计**：只编写让测试通过的最少代码（YAGNI）
2. **即时反馈**：每一步都有明确的验证标准
3. **文档即代码**：测试用例就是最好的文档
4. **重构安全网**：后续重构时有测试保护

### 何时跳过 TDD

- 原型/实验性代码（明确标注）
- 简单的配置变更
- 文档更新
- 用户明确要求不写测试

## 模式 4: 规格覆盖自检

**来源**: superpowers-zh writing-plans  
**适用场景**: 任务拆解完成后、进入实施前

### 执行步骤

1. 浏览规格文档（DESIGN.md / PROPOSAL.md / specs/）中的每个章节/需求
2. 指出实现该需求的任务编号
3. 列出所有遗漏的需求
4. 补充缺失任务

### 覆盖率检查表模板

```
| 规格需求 | 对应任务 | 覆盖？ |
|---------|---------|--------|
| Capability 1: 用户注册 | T005, T006, T007 | ✓ |
| Capability 2: 邮箱验证 | T008, T009 | ✓ |
| Capability 3: 密码重置 | — | ✗ 缺失！需补充 T010-T015 |
```

### 自检问题

- [ ] 规格中的每个需求都有对应任务？
- [ ] 任务的优先级与规格中的优先级一致？
- [ ] 边界条件和非功能需求（性能、安全）有对应任务？
- [ ] 测试策略中的每种测试类型都有对应任务？

## 模式 5: 类型一致性检查

**来源**: superpowers-zh writing-plans  
**适用场景**: 任务拆解完成后

### 检查内容

后续任务中使用的类型、方法签名和属性名是否与前面任务中定义的一致？

### 常见不一致问题

| 问题类型 | 示例 | 影响 |
|---------|------|------|
| 函数名不一致 | 任务 3 用 `clearLayers()`，任务 7 用 `clearFullLayers()` | 实现时混淆 |
| 类型名不一致 | 任务 2 定义 `UserDto`，任务 5 引用 `UserData` | 编译错误 |
| 参数顺序不一致 | 任务 3 定义 `createUser(email, name)`，任务 5 调用 `createUser(name, email)` | 运行时错误 |
| 属性名不一致 | 任务 2 定义 `userId`，任务 6 使用 `user_id` | 数据映射失败 |

### 检查步骤

1. 提取所有任务中引用的函数签名、类型定义、属性名
2. 与 DESIGN.md 中的接口定义对比
3. 发现不一致立即修正

## 模式 6: 执行交接

**来源**: superpowers-zh writing-plans  
**适用场景**: 计划完成后，移交执行阶段

### 两种执行方式

#### 方式 1: 子代理驱动（推荐）

- **适用场景**：复杂任务、多任务并行、需要审查
- **执行命令**：`implementation-build`
- **特点**：
  - 每个任务派发新子代理
  - 三层审查自动执行（语法、逻辑、规格对齐）
  - 任务间独立，失败不影响其他任务

#### 方式 2: 内联执行

- **适用场景**：简单任务、调试场景、快速原型
- **执行方式**：在当前会话中逐任务执行
- **特点**：
  - 上下文连续，便于调试
  - 适合 5 个以下的简单任务
  - 失败需要手动干预

### 交接模板

```markdown
## 执行交接

计划已完成并保存到 `specforge/changes/<ChangeName>/TASKS.md`。

**执行概览**：
- 总任务数：<N>
- 预估总时长：<M> 分钟
- 可并行任务：<K> 个
- 复杂度分布：S(<n>) / M(<m>) / L(<l>) / XL(<xl>)

**两种执行方式**：

1. **子代理驱动（推荐）** - 使用 `implementation-build` 命令
   - 每个任务调度新的子代理
   - 任务间进行三层审查
   - 快速迭代，失败隔离

2. **内联执行** - 在当前会话中使用 executing-plans
   - 批量执行并设有检查点
   - 适合简单任务或调试

**选哪种方式？**
```

## 使用指南

在 planning-breakdown 命令执行过程中，根据当前步骤引用对应的模式：

- **Step 2（文件结构先行）** → 参考模式 1
- **Step 3（小步骤任务拆解）** → 参考模式 2、3
- **Step 6（占位符扫描）** → 参考模式 2 的"任务描述要求"
- **Step 7（规格覆盖自检）** → 参考模式 4、5
- **Step 9（执行交接）** → 参考模式 6
