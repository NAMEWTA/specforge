# Complexity Guide（复杂度分析与扩展指南）

> 本指南定义了复杂度分析和任务扩展机制，用于评估任务难度并在必要时拆解高复杂度任务。

---

## 一、复杂度评级标准

### 1.1 四级复杂度模型

| 级别 | 标准 | 处理策略 | 预估时长 | 示例 |
|------|------|---------|---------|------|
| **S（简单）** | 单一文件、单一关注点、无外部依赖 | 正常执行 | 5-15 分钟 | 添加配置项、修改常量、更新文档 |
| **M（中等）** | 多文件、单一模块、有外部依赖 | 注意审查 | 15-60 分钟 | 新增服务层、修改 API、添加新功能 |
| **L（复杂）** | 跨模块、多依赖、需数据迁移 | 标注 [COMPLEX]，建议拆解 | 1-4 小时 | 重构核心模块、引入新框架、性能优化 |
| **XL（极复杂）** | 跨系统、架构变更、多团队协作 | 必须拆解为 2+ 个子任务 | 4+ 小时 | 微服务拆分、数据库迁移、架构升级 |

### 1.2 复杂度评估维度

评估任务复杂度时，考虑以下维度：

#### 维度 1: 范围（Scope）
- **S**: 单个文件内的变更
- **M**: 同一模块内多个文件
- **L**: 跨多个模块
- **XL**: 跨系统/服务

#### 维度 2: 依赖（Dependencies）
- **S**: 无外部依赖
- **M**: 1-2 个外部依赖（库、服务）
- **L**: 3-5 个外部依赖，或需要数据迁移
- **XL**: 5+ 外部依赖，或需要协调多个团队

#### 维度 3: 风险（Risk）
- **S**: 影响范围小，易于回滚
- **M**: 影响局部功能，回滚成本中等
- **L**: 影响核心功能，回滚成本高
- **XL**: 影响系统稳定性，回滚复杂

#### 维度 4: 不确定性（Uncertainty）
- **S**: 方案明确，无未知因素
- **M**: 方案基本明确，少量未知
- **L**: 方案需要探索，多个未知
- **XL**: 需要原型验证，大量未知

### 1.3 复杂度分布检查

生成任务列表后，检查复杂度分布：

```
✅ 健康分布：
S: 40-60%（大量简单任务，保证进展）
M: 30-40%（中等任务，核心价值）
L: 10-20%（复杂任务，标注并拆解）
XL: 0-10%（极复杂任务，必须拆解）

⚠️ 警告信号：
- L+XL > 30%：设计可能过于复杂，考虑重新设计
- 所有任务都是 S：粒度可能太细，考虑合并
- 没有 M 任务：可能缺少核心功能实现
```

## 二、任务扩展机制

### 2.1 何时使用任务扩展

当任务满足以下任一条件时，使用任务扩展：

- 复杂度评级为 **L（复杂）** 或 **XL（极复杂）**
- 预估时长超过 **2 小时**
- 涉及 **3+ 个模块** 的变更
- 需要 **数据迁移** 或 **架构变更**

### 2.2 扩展参数

任务扩展需要以下参数：

```json
{
  "subtaskCount": 5,
  "task": {
    "id": "T015",
    "title": "重构用户认证模块",
    "description": "将现有的 session-based 认证迁移到 JWT 认证"
  },
  "nextSubtaskId": 45,
  "additionalContext": "需要保持向后兼容，支持双认证模式过渡",
  "complexityReasoningContext": "跨 3 个模块（auth、middleware、api），涉及数据迁移（session 表），预估 6 小时",
  "hasCodebaseAnalysis": true,
  "projectRoot": "/path/to/project"
}
```

#### 参数说明

| 参数 | 类型 | 必需 | 说明 |
|------|------|------|------|
| subtaskCount | number | 是 | 要生成的子任务数量 |
| task | object | 是 | 父任务对象（id, title, description） |
| nextSubtaskId | number | 是 | 子任务起始 ID（必须连续） |
| additionalContext | string | 否 | 额外上下文信息 |
| complexityReasoningContext | string | 否 | 复杂度分析理由 |
| gatheredContext | string | 否 | 收集的项目上下文 |
| hasCodebaseAnalysis | boolean | 否 | 是否有代码库分析 |
| projectRoot | string | 否 | 项目根路径 |

### 2.3 子任务 JSON 结构

扩展后的子任务必须包含以下字段：

```json
{
  "subtasks": [
    {
      "id": 45,
      "title": "创建 JWT 工具模块",
      "description": "实现 JWT token 生成、验证、刷新功能",
      "dependencies": [],
      "details": "在 src/utils/jwt.ts 中实现：generateToken()、verifyToken()、refreshToken() 三个核心函数，使用 jsonwebtoken 库",
      "status": "pending",
      "testStrategy": "单元测试覆盖三个核心函数，包括正常场景和过期/无效 token 场景"
    },
    {
      "id": 46,
      "title": "修改认证中间件支持 JWT",
      "description": "扩展现有中间件，同时支持 session 和 JWT 认证",
      "dependencies": [45],
      "details": "在 src/middleware/auth.ts 中添加 JWT 验证逻辑，通过请求头 Authorization: Bearer <token> 识别认证方式",
      "status": "pending",
      "testStrategy": "集成测试验证双认证模式，确保向后兼容"
    }
  ]
}
```

#### 子任务字段说明

| 字段 | 类型 | 必需 | 最小长度 | 说明 |
|------|------|------|---------|------|
| id | number | 是 | - | 必须从 nextSubtaskId 开始连续递增 |
| title | string | 是 | 5-200 字符 | 清晰的子任务标题 |
| description | string | 是 | 10+ 字符 | 详细的子任务描述 |
| dependencies | array | 是 | - | 依赖的子任务 ID 数组（可为空） |
| details | string | 是 | 20+ 字符 | 实现细节（具体文件、函数、算法） |
| status | string | 是 | - | 必须为 "pending"（新任务） |
| testStrategy | string | 否 | - | 测试策略（可为 null） |

### 2.4 任务扩展示例

#### 原始任务（复杂度 L）

```
- [ ] T015 [COMPLEX] 重构用户认证模块
  - 从 session-based 迁移到 JWT
  - 涉及文件：src/middleware/auth.ts, src/services/auth-service.ts, src/utils/session.ts
  - 预估：6 小时
```

#### 扩展后的子任务

```
## 任务扩展：T015 → T045-T052

### Phase 1: 基础设施（T045-T047）
- [ ] T045 [P] 安装 JWT 依赖: `pnpm add jsonwebtoken @types/jsonwebtoken`
- [ ] T046 [P] 创建 JWT 配置 in src/config/jwt.ts
  - 密钥管理、过期时间、算法选择
- [ ] T047 创建 JWT 工具模块 in src/utils/jwt.ts
  - generateToken()、verifyToken()、refreshToken()
  - 测试：tests/unit/utils/jwt.test.ts

### Phase 2: 认证层改造（T048-T050）
- [ ] T048 修改认证中间件 in src/middleware/auth.ts
  - 支持双认证模式（session + JWT）
  - 通过 Authorization 头识别认证方式
- [ ] T049 扩展认证服务 in src/services/auth-service.ts
  - 添加 loginWithJWT()、refreshToken() 方法
  - 保留现有 loginWithSession() 向后兼容
- [ ] T050 编写集成测试 in tests/integration/auth.test.ts
  - 测试 session 认证（现有功能）
  - 测试 JWT 认证（新功能）
  - 测试双认证模式共存

### Phase 3: 迁移与清理（T051-T052）
- [ ] T051 创建迁移脚本 in scripts/migrate-session-to-jwt.ts
  - 将活跃 session 转换为 JWT token
  - 测试：在 staging 环境验证
- [ ] T052 清理 session 相关代码
  - 移除 src/utils/session.ts
  - 更新文档
  - 最终测试：`pnpm test`
```

## 三、高复杂度任务拆解策略

### 3.1 垂直拆解（按功能）

**适用场景**：功能模块包含多个独立子功能

```
原始任务：实现用户管理模块（复杂度 L）

拆解：
- T001: 用户创建（含测试）
- T002: 用户查询（含测试）
- T003: 用户更新（含测试）
- T004: 用户删除（含测试）
- T005: 用户列表分页（含测试）
```

### 3.2 水平拆解（按层次）

**适用场景**：任务涉及多个技术层次

```
原始任务：实现数据导出功能（复杂度 L）

拆解：
- T001: 数据层 - 导出查询逻辑
- T002: 服务层 - CSV/Excel 格式转换
- T003: API 层 - 导出端点
- T004: 测试层 - 集成测试
```

### 3.3 阶段拆解（按时间）

**适用场景**：需要分阶段交付的大型变更

```
原始任务：数据库迁移（复杂度 XL）

拆解：
- Phase 1: 双写模式（新旧表同时写入）
- Phase 2: 数据迁移（历史数据迁移到新表）
- Phase 3: 读切换（读取切换到新表）
- Phase 4: 清理（移除旧表和兼容代码）
```

### 3.4 并行拆解（按独立性）

**适用场景**：任务间无依赖，可并行执行

```
原始任务：多页面 UI 重构（复杂度 L）

拆解：
- T001 [P]: 首页重构（独立）
- T002 [P]: 设置页重构（独立）
- T003 [P]: 个人中心重构（独立）
- T004: 导航栏整合（依赖 T001-T003）
```

## 四、代码库分析集成

### 4.1 分析优先原则

在拆解复杂任务前，先进行代码库分析：

```bash
# 1. 查找相关文件
find src -name "*.ts" | grep -E "(auth|session|jwt)"

# 2. 搜索相关代码
grep -r "session" src/ --include="*.ts" -l

# 3. 查看依赖关系
grep -r "import.*session" src/ --include="*.ts"
```

### 4.2 分析结果应用

基于代码库分析结果：

- 识别现有代码模式和约定
- 确定需要修改的文件范围
- 发现潜在的依赖冲突
- 评估测试覆盖现状
- 制定与现有代码集成的策略

### 4.3 代码库分析检查清单

- [ ] 已识别所有需要修改的文件
- [ ] 已理解现有代码模式和约定
- [ ] 已发现潜在依赖冲突
- [ ] 已评估测试覆盖现状
- [ ] 已制定集成策略

## 五、复杂度评估工作流

### 5.1 评估步骤

```
Step 1: 初步评估
  ├─ 涉及几个文件？
  ├─ 涉及几个模块？
  ├─ 有外部依赖吗？
  └─ 有数据迁移吗？

Step 2: 复杂度评级
  ├─ S: 单一文件，无依赖
  ├─ M: 多文件，单模块
  ├─ L: 跨模块，多依赖
  └─ XL: 跨系统，架构变更

Step 3: 分布检查
  ├─ L+XL > 30%？→ 重新设计
  ├─ 全是 S？→ 粒度太细
  └─ 健康分布？→ 继续

Step 4: 任务扩展（如需要）
  ├─ 复杂度 L/XL → 使用扩展机制
  ├─ 定义子任务参数
  └─ 生成依赖有序的子任务列表
```

### 5.2 自动化建议

对于 SpecForge CLI，可以考虑未来实现自动化复杂度评估：

```bash
# 未来可能的命令
specforge analyze-complexity --tasks=TASKS.md
# 输出：
# T001: S (单一文件)
# T002: M (3 个文件，1 个依赖)
# T003: L [COMPLEX] - 建议拆解为 T003a-T003c
```

## 六、显式评分公式（自动评分）

> 公式可被脚本/agent 机械执行，避免主观差异。

### 6.1 三个输入因子

| 因子 | 取值范围 | 含义 |
|------|---------|------|
| **D**（依赖数） | 0..N | 任务依赖的其它任务数（来自 TASKS.md DAG） |
| **M**（模块跨度） | 1..N | 任务触及的模块/包/服务数（粗略：触及目录数） |
| **U**（未知风险） | 0..3 | 0=完全已知，1=有 1 处未知技术点，2=方案需探索，3=需原型验证 |

### 6.2 复杂度分数

```
score = D + 2 * (M - 1) + 3 * U
```

权重含义：
- 依赖每增加 1 → +1（线性）
- 模块每多跨 1 个 → +2（耦合代价）
- 每多 1 级未知风险 → +3（探索成本最高）

### 6.3 分数 → 复杂度等级

| score | 等级 | 处理 |
|-------|------|------|
| 0-1 | **S** | 直接执行 |
| 2-4 | **M** | 直接执行（≤ 1h） |
| 5-8 | **L** | 标注 [COMPLEX]，建议拆解 |
| ≥ 9 | **XL** | **必须**拆解为 ≥ 2 子任务（不可直接执行） |

### 6.4 工作流示例

```
任务 T015：实现用户认证模块
- D = 2（依赖 T010 数据库 schema、T011 用户模型）
- M = 3（src/auth/、src/middleware/、src/api/）
- U = 1（JWT 库选型未定）

score = 2 + 2*(3-1) + 3*1 = 2 + 4 + 3 = 9 → XL，必须拆解
```

### 6.5 评分自检

- [ ] D 是否对照 TASKS.md DAG 计算？
- [ ] M 是否实际遍历过涉及目录而非凭印象？
- [ ] U 是否有具体的"未知点"列表？无未知点的任务必须 U=0
- [ ] XL 任务是否已应用拆解策略（垂直 / 水平 / 时序 / 并行）？

### 6.6 反规避

| 借口 | 现实 |
|------|------|
| "公式太机械，看感觉就行" | 凭感觉的评估在压力场景下倾向低估 30%-60% |
| "U 一律给 0 反正最后都能查" | 未知 = 风险源，刻意低估只会推迟修复时机 |
| "M 算 1 因为只改了一个文件" | M 是模块/职责数而非文件数，跨层调用算多模块 |
| "score=8 还能直接做" | 8 已属 L，应至少标注 [COMPLEX] 提醒下游审查者 |
