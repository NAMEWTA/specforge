# 反规避借口表

> 本文件系统化整理复盘阶段常见的合理化借口及其反驳。借鉴 superpowers-zh writing-skills 的反模式方法论。

## 复盘阶段常见借口

| 借口 | 现实 |
|------|------|
| "复盘太花时间，直接开始下一个任务" | 不复盘的团队会在同一个坑里跌倒两次。经验不沉淀，本次 change 的价值就损失了一半。 |
| "这次没什么好总结的，很顺利" | 顺利的原因本身就是有价值的经验。不记录下来，下次可能就忘了为什么顺利。 |
| "learnings.md 以后有空再写" | "以后"永远不存在。现在不写，永远不会被写。决策时的情境记忆会在 48 小时后模糊 80%。 |
| "技能优化太复杂，先用着再说" | 每次不用技能优化造成的摩擦损失会在 10 次使用后超过一次性优化的成本。15 分钟测试省下数小时。 |
| "报告随便写写就行，没人看" | 复盘报告是下次 change 的起点，质量决定起点高度。报告中的改进项会直接落人到配置和技能中。 |
| "指标收集太麻烦，估算就行" | 不准确的数据会导致错误的调优决策。自动化脚本只需运行一次，数据是客观的。 |
| "ADR 以后补" | 决策时的情境记忆会在 48 小时后模糊 80%。现在不记录决策理由和替代方案，以后就只剩结果了。 |
| "技能 TDD 测试太繁琐" | 未测试的技能总有问题。部署未测试的技能比后面修复浪费更多时间。 |
| "配置调优不重要，默认就好" | 默认配置不适合你的团队。调优是持续改进的核心，每次 change 都应该让下一次更顺畅。 |
| "经验只写在自己脑子里" | 不沉淀的经验等于没有经验。团队无法复用，下次遇到同样问题还要重新摸索。 |
| "这个 change 太小，不值得复盘" | 小 change 的摩擦点累积起来就是大问题。轻量复盘只需 15 分钟，记录 1-2 个核心经验即可。 |
| "技能 description 触发词够用了" | 够不够用要看实际匹配率。如果技能在本次 change 中未被触发但应该被触发，就需要优化。 |
| "constitution 规则太多了" | 每条规则都是从真实的痛苦中沉淀的。规则多说明团队经历多，不是规则的问题，是缺少预防机制。 |
| "错误字典没必要，看错误信息就行" | 错误字典提供标准化的错误编号和纠偏建议。统一的错误体系让复盘数据可追踪、可分析。 |
| "handoffs 配置随便写写" | handoffs 决定了阶段间的衔接质量。配置不当会导致阶段产物不符合下游需要，引发返工。 |

## 红线列表

以下行为触发"删除重来"：

### 复盘流程红线
- 跳过复盘直接开始新 change
- 经验教训写成空话（无具体场景、无可执行建议）
- 复盘报告评分全部给 5 分（未真实反思）
- 指标全部估算（不使用自动化脚本收集）

### 经验沉淀红线
- ADR 缺少"决策理由"和"替代方案"
- learnings.md 只写"做得好的"，不写"需要改进的"
- 术语定义包含循环引用（A 定义为 B，B 定义为 A）
- 改进项未标注落点位置（config/skills/references）

### 技能优化红线
- 技能优化不经过基线测试
- 技能 description 总结工作流（导致 Agent 跳过正文）
- 未测试的技能直接部署
- 技能长度超过 500 行且未拆分到 references/

### 项目调优红线
- 配置变更未验证（直接提交）
- 触发词调整未测试匹配率
- Iron Law 新增未标注来源和触发场景
- 错误字典缺少 severity 或 guidance 字段

**以上所有都意味着：停下来，重新执行正确的流程。**

## 借口分类体系

### 时间压力类
**特征**: 强调"没时间"、"以后再做"
**应对策略**: 
- 提供轻量级替代方案（15 分钟复盘）
- 强调"以后"的代价（记忆模糊 80%、返工成本）
- 自动化降低时间成本（脚本收集指标）

**示例借口**:
- "复盘太花时间"
- "以后有空再写"
- "指标收集太麻烦"

### 过度自信类
**特征**: 强调"很简单"、"没问题"、"我有信心"
**应对策略**:
- 提供真实案例（返工次数、缺陷密度）
- 强调"简单"的欺骗性（简单逻辑也会出错）
- 要求数据支撑（"用数据证明很简单"）

**示例借口**:
- "这次没什么好总结的"
- "报告随便写写就行"
- "技能 TDD 测试太繁琐"

### 责任转移类
**特征**: 强调"没人看"、"不重要"、"默认就好"
**应对策略**:
- 明确责任归属（"下次 change 的起点"）
- 强调累积效应（"10 次使用后超过优化成本"）
- 提供具体影响（"错误的调优决策"）

**示例借口**:
- "没人看报告"
- "配置调优不重要"
- "经验只写在自己脑子里"

### 技术回避类
**特征**: 强调"太复杂"、"够用就行"
**应对策略**:
- 拆解复杂度（分步骤、提供模板）
- 提供量化标准（"匹配率 < 50% 需优化"）
- 强调长期收益（"持续改进的核心"）

**示例借口**:
- "技能优化太复杂"
- "触发词够用了"
- "错误字典没必要"

## 反规避语言模式

### 模式 1: 明确堵住漏洞

**不要只是陈述规则——禁止具体的变通方法**:

❌ **差**:
```markdown
跳过复盘？这是不允许的。
```

✅ **好**:
```markdown
跳过复盘直接开始新 change？触发 Iron Law 阻断，删除新 change 的产物，先完成复盘。

**无例外:**
- 不要保留作为"参考"
- 不要"简单记录几点"就算复盘
- 不要看新 change 的内容
- 删除就是删除
```

### 模式 2: 应对"精神 vs 字面"的辩论

**在前面加入基础原则**:

```markdown
**违反规则的字面意思就是违反规则的精神。**

"简单记录几点" ≠ 复盘。复盘需要结构化数据收集、经验沉淀、技能优化和项目调优。
缺少任何一环，都不叫复盘。
```

### 模式 3: 构建系统化借口表

从复盘中捕获借口，持续更新本文件。每个借口必须：
- **原文引用**（不要改写）
- **提供具体反驳**（包含数据或案例）
- **标注分类**（时间压力/过度自信/责任转移/技术回避）

## 持续更新

本文件是活的文档，每次复盘后应：

1. **捕获新借口**: 从本次 change 中识别新的合理化借口（原文引用）
2. **添加反驳**: 基于真实案例和数据编写反驳
3. **分类标注**: 标注借口分类（时间压力/过度自信/责任转移/技术回避）
4. **更新红线**: 如果借口导致了严重后果，添加到红线列表

**目标**: 让借口表成为团队的反模式知识库，帮助每次 change 都更顺畅。
