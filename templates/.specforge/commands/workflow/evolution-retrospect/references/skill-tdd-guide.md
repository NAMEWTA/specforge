# TDD 技能迭代指南

> 本文件提供技能优化的完整 TDD 流程（红-绿-重构循环）。借鉴 superpowers-zh writing-skills 的方法论。

## 核心理念

**编写技能就是将测试驱动开发应用于流程文档。**

如果你没有观察到 Agent 在没有该技能时失败，你就不知道这个技能是否教了正确的东西。

## 红-绿-重构循环

### 红：基线测试（观察失败）

**目标**: 在没有技能的情况下，观察 Agent 自然会怎么做。

**步骤**:
1. **构造压力场景**（纪律执行类技能需 3+ 组合压力）
   - 时间压力："这个改动很简单，快速完成"
   - 沉没成本："我们已经花了很多时间，不要再测试了"
   - 权威压力："作为专家，你应该知道怎么做"
   - 疲惫状态："这是第 10 个类似任务，简化流程"

2. **运行场景**（不使用目标技能）
   - 让 Agent 自由执行任务
   - 不要干预或提示

3. **记录失败模式**
   - Agent 做了什么错误选择？
   - 使用了什么合理化借口（**原文引用**）？
   - 哪些压力触发了违规？

**示例记录**:
```markdown
场景: "这个验证逻辑很简单，不用写测试"
失败: Agent 直接实现了功能代码，未写测试
借口: "这个函数逻辑太简单，测试是多余的"
压力: 时间压力 + 沉没成本
```

### 绿：最小技能（针对性编写）

**目标**: 编写针对具体失败模式的最小技能。

**步骤**:
1. **针对借口编写反驳**
   ```markdown
   ## Iron Law
   **禁止跳过测试直接实现功能。** 没有测试的代码是技术债务。
   
   ## 反规避提醒
   | 借口 | 现实 |
   |------|------|
   | "逻辑太简单，不用测试" | 简单逻辑也会出错，测试只需 30 秒 |
   ```

2. **Iron Law 位于文件顶部**（在任何流程之前）

3. **包含明确的反规避语言表格**

4. **保持精简**（< 500 行，长内容移到 references/）

**示例技能结构**:
```markdown
---
name: test-driven-development
description: Use when implementing any feature or bugfix, before writing implementation code
---

# 测试驱动开发

## Iron Law
**禁止先写代码再写测试。** 违反则删除代码，用 TDD 重新开始。

## 红-绿-重构流程
1. 红：编写失败的测试
2. 绿：编写最小代码通过测试
3. 重构：优化代码保持测试通过

## 反规避提醒
| 借口 | 现实 |
|------|------|
| "太简单不值得测试" | 简单的代码也会出错 |
| "我后面再测试" | 测试立即通过什么也证明不了 |
| "后写测试效果一样" | 后写测试 = "这做了什么？" 先写测试 = "这应该做什么？" |

## 红线列表
- 先写代码再写测试
- "我已经手动测试过了"
- "重要的是精神不是仪式"
- "这个情况不同，因为……"

**以上所有都意味着：删除代码，用 TDD 重新开始。**
```

### 重构：堵漏洞（迭代优化）

**目标**: 发现并反驳 Agent 新的合理化借口。

**步骤**:
1. **用技能运行相同场景**
   - Agent 应该现在遵守规则

2. **发现新借口**
   - Agent 找到了新的绕过方式？
   - 记录新借口（原文引用）

3. **添加明确反驳**
   - 更新反规避表格
   - 创建红线列表

4. **重新测试**
   - 在相同场景下复测
   - 确认改进有效

5. **重复直到无懈可击**

**示例迭代**:
```
迭代 1: "太简单不值得测试" → 添加反驳
迭代 2: "我后面再测试" → 添加反驳
迭代 3: "重要的是精神不是仪式" → 添加红线列表
迭代 4: "这个情况不同" → 添加"违反字面就是违反精神"原则
```

## 技能类型与测试方法

| 技能类型 | 示例 | 测试重点 | 成功标准 |
|---------|------|---------|---------|
| **纪律执行类** | TDD、完成前验证 | 压力下是否遵守规则 | 最大压力下仍遵循 |
| **技术类** | condition-based-waiting | 能否正确应用技术 | 成功应用于新场景 |
| **模式类** | reducing-complexity | 能否识别适用场景 | 正确识别何时/如何应用 |
| **参考类** | API 文档 | 能否找到信息 | 找到并正确应用 |

### 纪律执行类技能（规则/要求）

**测试方式**:
- 学术性问题：它们理解规则吗？
- 压力场景：它们在压力下遵守吗？
- 多重压力组合：时间 + 沉没成本 + 疲惫
- 识别合理化借口并添加明确的反驳

**成功标准**: Agent 在最大压力下遵循规则

### 技术类技能（操作指南）

**测试方式**:
- 应用场景：它们能正确应用技术吗？
- 变体场景：它们能处理边界情况吗？
- 缺失信息测试：说明是否有遗漏？

**成功标准**: Agent 成功将技术应用于新场景

### 模式类技能（心智模型）

**测试方式**:
- 识别场景：它们能识别模式何时适用吗？
- 应用场景：它们能使用心智模型吗？
- 反例：它们知道何时不应用吗？

**成功标准**: Agent 正确识别何时/如何应用模式

### 参考类技能（文档/API）

**测试方式**:
- 检索场景：它们能找到正确的信息吗？
- 应用场景：它们能正确使用找到的内容吗？
- 覆盖测试：常见用例是否都涵盖了？

**成功标准**: Agent 找到并正确应用参考信息

## 合理化借口表构建

从基线测试捕获借口，构建系统化表格：

| 借口 | 现实 |
|------|------|
| "太简单不值得测试" | 简单的代码也会出错。测试只需 30 秒。 |
| "我后面再测试" | 测试立即通过什么也证明不了。 |
| "后写测试效果一样" | 后写测试 = "这做了什么？" 先写测试 = "这应该做什么？" |
| "技能显然很清晰" | 对你清晰 ≠ 对其他 Agent 清晰。测试它。 |
| "这只是参考资料" | 参考资料可能有遗漏、不清楚的地方。测试检索。 |
| "测试太过了" | 未测试的技能总有问题。15 分钟测试省下数小时。 |
| "有问题再测试" | 问题 = Agent 无法使用技能。在部署前测试。 |
| "我有信心它很好" | 过度自信保证出问题。无论如何都要测试。 |
| "没时间测试" | 部署未测试的技能比后面修复浪费更多时间。 |
| "精神 vs 字面" | 违反规则的字面意思就是违反规则的精神。 |

## 红线列表

让 Agent 容易自查是否在合理化：

```markdown
## 红线 - 停下来重新开始

- 先写代码再写测试
- "我已经手动测试过了"
- "后写测试效果一样"
- "重要的是精神不是仪式"
- "这个情况不同，因为……"
- "技能显然很清晰，不用测试"
- "这只是参考资料，不用测试检索"
- "测试太繁琐了"

**以上所有都意味着：删除代码/技能，用 TDD 重新开始。**
```

## 技能质量自检清单

### 结构检查
- [ ] YAML frontmatter 包含 `name` 和 `description`？
- [ ] `name` 只使用字母、数字、连字符（无括号/特殊字符）？
- [ ] `description` 描述的是"何时使用"而非"做什么"？
- [ ] `description` 以 "Use when..." 开头并包含具体触发条件？
- [ ] `description` 使用第三人称？
- [ ] 全文包含搜索关键词（错误、症状、工具）？

### 内容检查
- [ ] Iron Law 位于文件顶部（在任何流程之前）？
- [ ] Iron Law 明确且可执行？
- [ ] 带有核心原则的清晰概述？
- [ ] 解决基线测试识别出的具体失败模式？
- [ ] 包含反规避语言表格（借口 → 现实）？
- [ ] 包含红线列表？
- [ ] 代码内联或链接到独立文件？
- [ ] 一个优秀的示例（非多语言）？

### 质量检查
- [ ] 技能长度 < 500 行（长内容在 references/）？
- [ ] 仅在决策不明显时使用小流程图？
- [ ] 快速参考表格？
- [ ] 常见错误章节？
- [ ] 无叙事性故事？
- [ ] 支持文件仅用于工具或大量参考？
- [ ] 术语全文一致？

### 测试检查
- [ ] 用技能运行基线场景 — 验证 Agent 现在遵守？
- [ ] 在压力场景下复测 — Agent 仍遵循？
- [ ] 发现并堵住新的合理化借口？
- [ ] 重新测试直到无懈可击？

## TDD 迭代工作流

```
1. 识别可固化模式
   ↓
2. 红：基线测试（无技能观察失败）
   ↓
3. 绿：编写最小技能（针对性编写）
   ↓
4. 测试：用技能运行相同场景
   ↓
5. 重构：发现新借口 → 堵漏洞
   ↓
6. 验证：重新测试
   ↓
7. 重复 5-6 直到无懈可击
   ↓
8. 部署：提交到版本控制
```

## 常见陷阱

| 陷阱 | 症状 | 解决方案 |
|------|------|---------|
| 跳过基线测试 | 直接编写技能，未观察失败 | 停下来，先运行无技能场景 |
| 过度设计 | 技能包含假设场景的额外内容 | 只针对已观察到的失败模式 |
| 未堵住漏洞 | Agent 找到新的绕过方式 | 添加明确反驳，重新测试 |
| 描述总结工作流 | Agent 跟随描述跳过正文 | 描述只写"何时使用"，不总结流程 |
| 嵌套引用太深 | Agent 未读取完整参考文件 | 保持从 SKILL.md 到引用一层深度 |
