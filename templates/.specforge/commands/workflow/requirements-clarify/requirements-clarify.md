# 需求澄清与方案探索

## Iron Laws

> **HARD-GATE 1: 禁止在方案批准前进入任何实现或详细设计。**
> 反驳：“先写着代码，方案后面再定” → 99% 的方案文档永远不会被补上。

> **HARD-GATE 2: 禁止在规格自检完成前请用户审查。**
> 带有 TODO/占位符/矛盾的规格会浪费用户审查时间并降低信任。

> **HARD-GATE 3: 禁止接受模糊的方案批准。**
> “还行”、“再看看”、“不太确定” = 未批准。必须使用驳斥草案法直到获得明确批准。

补充门禁：遵循 `.specforge/config.yaml` 的 requirements 规则；若缺少可度量验收标准或存在未闭环澄清项，触发 **E002_unapprovedSolution** 阻断。

## 核心交互原则

1. **一次只问一个问题** — 每条消息只提一个问题，不要同时抛出多个
2. **每个问题附带推荐答案** — 让用户有东西可以回应，而不是面对空白输入
3. **先深挖再横向展开** — 得到回答后，先沿着这条线追到底，再转向新分支
4. **能调查就先调查** — 如果问题可以通过读代码/文件回答，先调查再提问
5. **当觉得“可以行动了”时，多问 3 个问题** — 那种“够了”的感觉只是表层
6. **不要把总结当推进** — “所以你的意思是 X、Y、Z”会结束盘问，而不是推进它
7. **对含糊回答保持追问** — “我以后再想”、“大概是 X”都意味着应该继续往下挖
8. **指出矛盾、闪躲和空话** — 语气可以礼貌，但不能软化到接受雾气本身

## 处理半截答案的技巧

当用户给出含糊回答（“我不确定”、“大概是 X”、“有点像 Y”）：

- **默认做法：给出可被反驳的草案**
  “我先给一个答案——你告诉我哪里不对：……”
  这比开放式逼问更有效，因为反驳比凭空创造更容易

- **如果用户质疑问题本身**：
  “要让这个问题可回答，你还需要知道什么？”
  然后沿着这条线继续追问

- **禁止行为**：
  - 不要接受雾气作为答案
  - 不要把“我以后再想”当成有效回答
  - 不要指出矛盾时软化语气到接受空话

## Phase 1: 上下文加载与意图挖掘

### 1.1 读取已有信息
- 读取 `specforge/project.md` — 项目上下文与技术栈
- 读取 `specforge/config.yaml` — 项目级机器源（context / rules / errors）
- 扫描 `specforge/brainstorming/` — 相关头脑风暴文档（如有）
- 读取 `specforge/context/` — 术语表、ADR、经验教训
- 扫描代码库结构，识别既有约束和可复用组件

### 1.2 识别抽象层级
- 当前需求属于哪个抽象层级？（基础设施/领域模型/应用服务/用户界面）
- 与已有代码的关系？（新增模块/修改既有/替换旧实现）

### 1.3 意图挖掘

**一次只问一个问题，每个问题附带推荐答案**：

- **意图与期望结果**：用户真正想要达成什么？表面需求背后的深层动机？
  - 示例：“对你来说，这个项目做成什么样你会觉得‘值了’？我猜可能是 X，对吗？”
  
- **约束暴露**：有哪些已知但未言明的约束？（时间、人力、技术栈、合规）
  - 示例：“如果只能保留一个特性，其他都可以砍，你选哪个？我猜是 P1 的核心流程？”

- **范围评估**：如果需求描述了多个独立子系统，立即指出这一点
  - 示例：“这个需求涉及 A、B、C 三个相对独立的部分。我们应该先聚焦哪个？我建议从 A 开始，因为它是基础。”

**Iron Law**: 不要跳过意图挖掘直接进澄清。没有清晰的意图，澄清会跑偏。

## Phase 2: 结构化需求澄清 — 6 维度分类法

**目标**：系统扫描需求中欠明确的区域。每个维度逐一审查，标记 `[NEEDS CLARIFICATION]`（最多 5 个）。

### 2.1 功能范围（Functional Scope）
- 核心功能边界在哪里？什么明确不在范围内？
- 用户角色有哪些？每个角色的权限和操作范围？
- 功能优先级（P1 必须 / P2 应该 / P3 锦上添花）？

### 2.2 领域与数据模型（Domain & Data Model）
- 核心实体有哪些？实体间关系？
- 数据从哪里来？流向哪里？
- 数据一致性要求？（强一致/最终一致）

### 2.3 交互与用户体验（Interaction & UX）
- 用户如何与此功能交互？（API/CLI/Web/Mobile）
- 关键用户流程的步骤？
- 错误状态和空状态如何处理？

### 2.4 非功能质量属性（Non-Functional Quality）
- 性能要求？（响应时间、吞吐量、并发数）
- 安全要求？（认证、授权、数据保护）
- 可用性要求？（SLA、故障恢复时间）

### 2.5 集成与依赖（Integration & Dependencies）
- 依赖哪些外部系统/服务？
- 接口协议和数据格式？
- 第三方服务的 SLA 和降级策略？

### 2.6 边界情况（Edge Cases）
- 极端输入如何处理？
- 并发冲突如何处理？
- 失败模式有哪些？如何恢复？

## Phase 3: 方案探索与对比

**目标**：通过多视角提问进行方案探索。提出 2-3 种可行方案，展示对比矩阵，给出推荐理由。

### 3.1 多视角提问（静默使用视角名称）

**核心视角（必问 3 个）**：
- **第一性原理**：如果不考虑任何现有方案，解决问题的最直接方式是什么？
- **隐藏假设挖掘**：方案建立在哪些未验证的假设上？
- **预先验尸**：假设项目失败，最可能的原因是什么？

**风险视角（选问 2-3 个）**：
- 钢人化反方：反对这个方案的最强论点是什么？
- 第二选择：如果不选推荐方案，最好的替代方案是什么？
- 可逆性：这是一个单向门还是双向门？做错后能多容易地回退？
- 可持续性：6 个月后维护这个方案的成本如何？

完整 12 视角参考：`references/multi-perspective-patterns.md`

### 3.2 理解用户意图与期望结果

- 一句话描述：用户希望达成什么？
- 期望的时间线和质量水平？
- 可接受的 trade-off 范围？

### 3.3 提出 2-3 种可行方案

每个方案必须包含：
- **思路**：一句话核心思路
- **关键决策**：2-3 个关键技术选择
- **适用场景**：此方案最适合什么情况
- **Trade-off**：主要优缺点（至少各 2 个）
- **估算**：大致工作量和风险水平

### 3.4 展示对比矩阵

| 维度 | 方案 A（推荐） | 方案 B | 方案 C |
|------|--------------|--------|--------|
| 思路 | ... | ... | ... |
| 复杂度 | 低/中/高 | 低/中/高 | 低/中/高 |
| 风险 | 低/中/高 | 低/中/高 | 低/中/高 |
| 开发成本 | 低/中/高 | 低/中/高 | 低/中/高 |
| 维护成本 | 低/中/高 | 低/中/高 | 低/中/高 |
| 可扩展性 | 低/中/高 | 低/中/高 | 低/中/高 |
| 团队匹配度 | 低/中/高 | 低/中/高 | 低/中/高 |

### 3.5 给出推荐理由

- 为什么推荐方案 A？
- 在什么条件下方案 B 会变成更好的选择？
- 方案 C 什么时候考虑？（如“项目规模扩大 3 倍后”）

### 3.6 视觉伴侣（当涉及视觉/布局/架构时）

**判断标准**: 用户看到它是否比读到它更容易理解？

**使用视觉伴侣**：
- UI 原型（线框图、布局、导航结构）
- 架构图（系统组件、数据流、关系图）
- 并排视觉对比（两种布局/设计方向）
- 空间关系（状态机、流程图、实体关系图）

**使用终端**：
- 需求和范围问题
- 概念性 A/B/C 选择
- 权衡列表、技术决策

详细指南：`references/brainstorming-visual-companion.md`

**Iron Law**: 不要为了用视觉伴侣而用。只有当视觉展示确实更容易理解时才使用。

## Phase 4: 方案批准与产物生成

### 4.1 方案审批 — HARD-GATE

> 这是整个 SpecForge 工作流中最重要的门禁。

向用户呈现推荐方案和对比矩阵，明确询问批准。**没有批准不能进入下一阶段。**

### 4.2 批准循环

```
提出推荐方案 → 用户反馈
  ├── 批准 → 进入 4.3
  ├── 模糊 → 驳斥草案法 → 提出具体草案 → 用户反馈 → 循环
  └── 反对 → 理解反对原因 → 调整方案 → 重新推荐 → 循环
```

**驳斥草案法**（处理模糊反馈）：
- 如果用户给出模糊反馈（“还行”、“再看看”、“不太确定”）
- 不要反复追问“你觉得怎么样”
- 先产生一个**可被驳斥的草稿**——具体的、有细节的方案描述
- 让用户批评一个具体的方案，比让用户描述期望的方案容易得多

**操作**：发现模糊反馈 → 立即生成具体草案 → “这个方案的具体设计是XXX，你觉得哪里需要调整？”

### 4.3 会话日志（可选但推荐）

在方案批准后、生成 PROPOSAL.md 之前，可选择性写入提炼后的会话日志：

```
specforge/changes/<YYYY-MM-DD-ChangeName>/CLARITY_LOG.md
```

结构：
- **意图**（用户真正想达成的目标）
- **约束**（浮现的不可谈判项）
- **关键决策**（决定了什么 + 为什么 + 考虑过的替代方案）
- **浮现出的假设**（从隐含到显式）
- **未决问题**（暂时回答不了，留到以后处理）
- **非目标**（明确选择不做的事情）

### 4.4 创建 Change 目录

方案批准后，创建 change 目录结构：

```bash
specforge/changes/<YYYY-MM-DD-ChangeName>/
├── .specforge.json        # change 元数据
└── PROPOSAL.md            # 需求与方案文档（见 4.5）
```

**ChangeName 命名规则**：简短英文描述，PascalCase 或 kebab-case。如 `NotificationCenter`、`user-auth-refactor`。

### 4.5 生成 PROPOSAL.md

#### 必需章节

```markdown
# PROPOSAL: <ChangeName>

## Why（动机）
- 为什么要做这个变更？
- 解决什么问题？

## What Changes（变更范围）
- 具体变更内容（新增/修改/删除）
- 明确不在范围内的内容
- 影响范围评估

## Approach（已批准方案）
- 选定的方案描述
- 关键技术决策
- 架构概览（高层）

## Alternatives Considered（已考虑方案）
| 方案 | 优势 | 劣势 | 决策 |
|------|------|------|------|
| ... | ... | ... | 采纳/不采纳，原因 |

## Capabilities（能力列表）
### P1（必须）
- [ ] Capability 1: <描述>
- [ ] Capability 2: <描述>

### P2（应该）
- [ ] Capability 3: <描述>

### P3（锦上添花）
- [ ] Capability 4: <描述>

## Impact（影响评估）
- 涉及模块/系统
- 数据迁移需求
- API 变更（向后兼容性）
- 安全考虑
- 性能影响
```

### 4.6 规格自检（生成 PROPOSAL.md 后必须执行）

以全新视角审视 PROPOSAL.md：

1. **占位符扫描** — 有没有“待定”、“TODO”、未完成的章节？修复它们
2. **内部一致性** — 各章节之间有矛盾吗？架构和功能描述匹配吗？
3. **范围检查** — 是否足够聚焦以用于单个实现计划？还是需要进一步拆分？
4. **模糊性检查** — 有没有需求可以被两种方式理解？选择一种并明确写出来
5. **YAGNI 检查** — 有没有未请求的功能、过度设计？移除它们

发现问题直接内联修复，无需重新审查。

### 4.7 规格审查（用户批准前）

在请用户审查 PROPOSAL.md 之前，可选择性调用规格审查员子代理：

参考：`references/spec-document-reviewer-prompt.md`

检查要点：
- 完整性（TODO、占位符、不完整的章节）
- 一致性（内部矛盾、冲突需求）
- 清晰度（模糊到可能构建出错误东西的需求）
- 范围（是否聚焦到单个计划，而非多个子系统）
- YAGNI（未请求的功能、过度设计）

### 4.8 用户审查关卡

规格自检完成后，请用户在继续之前审查书面规格：

> “规格已编写并保存到 `specforge/changes/<YYYY-MM-DD-ChangeName>/PROPOSAL.md`。请审查一下，如果在我们开始编写实现计划之前你想做任何修改，请告诉我。”

等待用户回复。如果他们要求修改，做出修改并重新运行规格自检。只有在用户批准后才继续。

**Iron Law**: 禁止在规格自检完成前请用户审查（HARD-GATE 2）。

### 4.9 完成衔接

**产物**：
- `specforge/changes/<YYYY-MM-DD-ChangeName>/` — 新变更目录
- `specforge/changes/<YYYY-MM-DD-ChangeName>/.specforge.json` — 变更元数据（phase: proposal, status: draft）
- `specforge/changes/<YYYY-MM-DD-ChangeName>/PROPOSAL.md` — 需求与方案文档（含需求澄清 + 方案探索 + 已批准方案）
- `specforge/changes/<YYYY-MM-DD-ChangeName>/CLARITY_LOG.md` — 会话日志（可选）

**下一步**：执行 `design-explore` 命令，基于已批准方案进行架构与技术设计。

**警告**：在进入 design-explore 之前，必须确认 PROPOSAL.md 中的方案已由用户明确批准。未经批准的方案进入设计阶段是 Iron Law 违规（HARD-GATE 1）。

## 反模式：不要这样做

- **\u201c这个太简单了，不需要澄清\u201d** — 简单项目恰恰是未经检验假设造成最多浪费的地方。一个待办事项列表、一个单函数工具、一个配置变更——全都需要。设计可以很简短（对于真正简单的项目几句话就够了），但你必须展示出来并获得批准。

- **把总结当推进** — \u201c所以你的意思是 X、Y、Z\u201d会结束盘问，而不是推进它。要继续提问，不要复述。

- **问题问得太少且太早收敛** — 刻意对抗这种倾向。当你觉得自己已经可以行动时，再多问三个问题。那种\u201c够了\u201d的感觉只是表层，不是底层。

- **不是找 bug** — 本阶段目标是扩展用户对\u201c想要什么、为什么想要\u201d的理解，不是排查执行层问题。你不是在查竞争条件、错误定位或薄弱的 SOP 步骤。

- **不是教练** — 不要鼓励，不要验证，不要安慰。要追问。

- **不是清单** — 没有强制问题、没有固定数量、没有固定顺序。根据用户刚刚说的话调整。

## 常见阻断与纠偏（errors 引用）

- **E001_missingPrerequisiteArtifact**：缺少 `specforge/config.yaml` 或 `specforge/project.md` → 回到 foundation-init 补齐。

- **E002_unapprovedSolution**：方案未批准 / 验收标准不可度量 / `[NEEDS CLARIFICATION]` 未闭环 → 禁止进入 design-explore。

- **E003_prematureImplementation**：方案批准前检测到代码生成/实现行为 → 立即阻断并提醒 HARD-GATE 1。

- **E004_incompleteSpecSelfReview**：规格自检未完成即提交用户审查 → 提醒 HARD-GATE 2 并返回自检。

## 反规避提醒

| 借口 | 现实 |
|------|------|
| \u201c需求很清楚，不需要澄清\u201d | 你以为清楚的需求在 2 周后会变成\u201c我说的不是这个意思\u201d |
| \u201c只有一个方案，不需要对比\u201d | \u201c只有一个方案\u201d = 思考不够深入。至少找出 2 个真正不同的方案 |
| \u201c先用推荐方案做着，不行再换\u201d | 架构级别的方案切换成本通常是初始实现成本的 3-5 倍 |
| \u201c方案文档后面补上\u201d | 99% 的方案文档永远不会被补上。不写下来 = 不存在 |
| \u201c这个太简单了，不需要设计\u201d | 简单项目恰恰是未经检验假设造成最多浪费的地方 |
