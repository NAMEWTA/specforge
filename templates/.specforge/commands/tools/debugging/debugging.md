# 系统化调试

关键原则:**找到根因前禁止提出修复方案。**

## Iron Law

> **没有根因调查,就没有修复方案。** 禁止跳过诊断直接猜修复。禁止在不确定根因的情况下"试一下这个改动"。

## 快速开始(你应该怎么做)

当用户报告问题时,按以下决策树执行:

```
问题出现
  │
  ├─ 有明确错误信息? ──是──→ 进入 Phase 1.1 (阅读错误)
  │
  ├─ 测试失败? ──是──→ 进入 Phase 1.2 (稳定复现)
  │                   └─→ 如需隔离污染测试 → 读取 references/root-cause-tracing.md
  │
  ├─ 性能问题? ──是──→ 进入 Phase 1.3 (检查近期变更)
  │                   └─→ 如涉及异步等待 → 读取 references/condition-based-waiting.md
  │
  └─ 异常行为(无报错)? ──→ 进入 Phase 1.4 (多组件证据收集)
                          └─→ 追踪数据流 → 读取 references/root-cause-tracing.md
```

**关键原则:**
- 在提出任何修复方案之前,必须先完成 Phase 1 (根因调查)
- 多组件系统 → 先添加诊断埋点收集证据,再分析
- 调用栈深处报错 → 使用根因追踪技术反向查找源头

**何时跳过本命令:**
- 问题已明确根因 → 直接实施修复
- 只是语法错误/拼写错误 → 直接修复,无需走完整流程

---

## Phase 1: 根因调查

### 1.1 问题复现
- [ ] 确定问题出现的准确条件(环境、输入、操作序列)
- [ ] 编写最小复现步骤(越短越好,理想 < 5 步)
- [ ] 确认问题可稳定复现(不是偶发)
- [ ] 捕获完整错误信息(消息、堆栈、日志、请求 ID)

### 1.2 多组件证据收集

在提出任何修复方案之前:
- 在涉及的每个组件边界添加诊断日志
- 运行一次复现用例收集所有边界的证据
- 分析证据,追踪数据从入口到出错点的完整路径
- **只收集一次证据,不反复猜测性修改代码**

**何时需要反向追踪:**
- 错误发生在调用栈深处
- 不清楚无效数据从哪里来
- 需要找到是哪个操作触发了问题

→ **深度技术:** 读取 `references/root-cause-tracing.md` 获取完整追踪流程

### 1.3 隔离可疑范围

- 二分法缩小范围(注释掉一半代码,问题是否还在?)
- 确认问题引入时间点(git bisect 如适用)
- 区分"触发点"和"根因点"(报错的地方 ≠ 问题起源的地方)

---

## Phase 2: 模式分析

### 2.1 错误模式识别
- 这是单点故障还是系统性问题?
- 错误是否与特定数据/时间/并发条件有关?
- 是否有相似的历史问题?(搜索 issue tracker、commit log)

→ **常见模式库:** 读取 `references/debugging-patterns.md` 对照识别

### 2.2 3 振出局规则(3-Strikes Rule)

> 如果连续 3 次修复尝试都失败了,**立即停止**。不要再尝试第 4 次修复。
> 3 次失败 = 你对问题的理解模型有问题。回到 Phase 1 重新调查。

### 2.3 Red Line 自检

如果出现以下想法,立即停止并返回 Phase 1:
- "试试改一下 X 看能不能好"
- "同时改几个地方,一次搞定"
- "不用加诊断日志,我大概知道问题在哪"
- "应该是 X 的问题"
- "之前遇到过类似的,改一下就行"

---

## Phase 3: 假设验证

### 3.1 明确假设

将你的猜测写成可验证的假设:
```
假设:问题由 <具体原因> 引起
验证方式:<具体测试或操作>
预期结果:<如果假设正确应该看到什么>
```

### 3.2 逐一验证
- 一次只验证一个假设
- 设计最小验证实验(只涉及必要的代码变更)
- 记录验证结果(确认/排除)
- 排除的假设也要记录(避免重复猜测)

**时序问题假设:** 如涉及异步等待、竞态条件
→ **条件等待技术:** 读取 `references/condition-based-waiting.md`

---

## Phase 4: 实施修复

### 4.1 设计修复方案(根因确认后)
- 修复是否解决了根因而不仅是症状?
- 修复的影响范围?(运行 `git diff --stat` 确认)
- 修复是否会引入新问题?

### 4.2 纵深防御(Defense in Depth)

在修复层添加测试(确保问题不会回归) → 在上层添加验证(确保即使底层出错也能被检测) → 在系统层添加监控(确保同类问题能被及时发现)

**四层防御策略:**
1. 入口校验(API 边界拒绝无效输入)
2. 业务逻辑校验(确保数据对当前操作合理)
3. 环境守卫(防止特定上下文的危险操作)
4. 调试埋点(记录上下文以便事后分析)

→ **完整指南:** 读取 `references/defense-in-depth.md`

### 4.3 验证修复
- [ ] 运行复现用例确认问题已解决
- [ ] 运行完整测试套件确认无回归
- [ ] 对修复代码本身做 Code Review
- [ ] 记录:根因 → 修复方案 → 预防措施

---

## 完成检查清单

- [ ] 根因已确认(不是猜测)
- [ ] 修复已通过测试验证
- [ ] 无回归问题
- [ ] 未触发 3-Strikes Rule(修复尝试 < 3 次)
- [ ] 防御层已加固

## 反规避提醒

| 借口 | 现实 |
|------|------|
| "我先试几个可能的修复" | 在没有诊断的情况下一通乱试 = 在黑暗中开枪。可能"修复"了症状但掩盖了根因 |
| "这个 bug 很简单,不用走完整流程" | 简单的 bug 往往有更深的根源。10 分钟调查 > 2 小时盲目修复 |
| "我之前遇到过类似的,直接改" | 模式相似 ≠ 根因相同。先验证,后修复 |
| "同时改几个地方省时间" | 无法隔离哪个生效了。还会引入新 bug |
| "跳过测试,我手动验证" | 手动验证不可重复。先写测试才能证明修复有效 |
